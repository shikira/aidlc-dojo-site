## Tri-Modeの質問フロー

AI-DLC には、ステージ中にエージェントとやり取りする方法が 3 つ用意されています。この **Tri-Mode(三モード)質問フロー** は、ステージが入力を集めるときに提示され、あなたはその場面に最も合うモードを選べます。

```
▸ Choose interaction mode:
  (1) Guide Me — agent asks structured questions
  (2) Edit File — write directly to the artifact
  (3) Chat — freeform discussion
```

**Guide Me(ガイドしてもらう)** は、エージェントが構造化された質問で一問ずつ(あるいはまとめて)導いてくれるモードです。エージェントに会話を主導させ、抜け漏れなく進めたいときに向きます。あなたは各質問に直接答え、回答はトレーサビリティのため、そのステージの質問ファイルに記録されます。

**Edit File(ファイルを編集)** は、エージェントが質問ファイルを作成(または開いて)、あなたが直接そこに書き込むモードです。すでに何を書きたいか分かっていて、質問に答えるより自分で書き下したいときに向きます。空欄の回答フィールドを持つ質問ファイルがレコードディレクトリに現れ、あなたが自分のペースで埋め、エージェントが完成したファイルを読んで先へ進みます。

**Chat(自由対話)** は、エージェントと自由に議論するモードです。要件がまだ固まっていない、アイデアを探りたいときに向きます。あなたは自由に話し、エージェントが会話から決定を抽出して、正の記録として質問ファイルへ書き戻します。

三つのモードはいずれも「質問ファイル」という一つの正本に収束します。だからステージの途中でモードを切り替えても進捗は失われず、すでに捉えた回答はファイルに残ります。なお、ハーネスによって見え方は異なります — Claude Code は `AskUserQuestion` ウィジェットを使い、Kiro と Codex は番号付きの散文で選択肢を出しますが、質問ファイルが正本である点、そして「いつゲートが発火し、何を尋ね、あなたが主導権を握る」という意味論は、エンジンに宿るため全ハーネス共通です。

三モードを使い分ける勘所は、あなたが「何をどれだけ分かっているか」にあります。要件が漠然としていて探索から始めたいなら Chat が向きます。エージェントとの自由な対話から、決定が少しずつ形になり、質問ファイルへ書き戻されていきます。逆に、作りたいものが既にはっきりしているなら Edit File が速い — 空欄のフィールドを自分のペースで埋め、エージェントは完成を待って読み取ります。抜け漏れなく網羅したいなら Guide Me が安心で、エージェントが構造化された問いで一つずつ導きます。実際のワークフローでは、Chat で発散し、Guide Me で詰め、Edit File で仕上げる、といったモード横断が自然に起こります。切り替えても正本は一つなので、迷ったらモードを変えてみる、という気軽さで使えるのが Tri-Mode の良さです。この「入力の仕方は選べるが、記録は一つに集約する」という設計が、後段のトレーサビリティと、人間が主導権を握り続ける土台を同時に支えています。

質問への回答は、そのステージの成果物へと結実します。たとえば Intent Capture(意図の把握)なら、`intent-capture.md`(問題文・対象ユーザー・成功基準)と `intent-capture-questions.md`(5 つの質問と、そのすべての回答)がレコードディレクトリに生まれます。ステージが済むと、エージェントは完了サマリを表として提示し、続けて承認ゲートを開きます。あなたはそこで作業を確認し、良ければ承認、直したければ変更を要求します。質問フローは単なる入力インターフェイスではなく、「何を尋ね、あなたがどう答えたか」を成果物として残し、後から追跡できるようにする仕組みなのです。だからこそ、どのモードを選んでも決定はトレース可能な形で蓄積され、人間の意図が記録として残ります。ここが、行き当たりばったりのプロンプト対話と、AI-DLC の構造化された質問フローとの決定的な違いです。

## 承認ゲートと3ストライク離脱

Initialization の 3 ステージを除くすべてのステージは、末尾に **承認ゲート** を持ちます。これは、ワークフローが前へ進む前に、あなたがエージェントの作業を確認するチェックポイントです。標準のゲートは二択を提示します。

```
▸ How would you like to proceed?
  (1) Approve — Continue to [next stage]
  (2) Request Changes — Provide revision feedback
```

**Approve(承認)** は、そのステージを完了として印を付け、`aidlc-state.md` を更新し、進捗行(例: `Progress: 13/32 overall | 3/7 IDEATION stages complete.`)を表示して次のステージへ進めます。**Request Changes(変更を要求)** は、具体的なフィードバックを渡すと、エージェントが作業を修正して承認ゲートを再提示します。

このゲートは、本物の人間の関与を要求します。プロンプトを打つ、あるいは質問ウィジェットに答えるという行為が監査台帳に「人間のターン(HUMAN_TURN)」として記録され、承認は前回のゲート解決以降にその記録がなければ拒否されます。自動操縦で走るモデルが、人間不在のまま承認をでっち上げることはできない仕組みです。ゲートが人間のターンを記録しないハーネスでは、一度「approve」などと短く打てば記録が残ります。

同じステージで変更要求が **3 回以上** 続くと、第三の選択肢 **Accept as-is(現状のまま受け入れる)** が現れます。これは現行版の成果物をアーカイブしてワークフローを前へ進めるもので、「完璧が善の敵」になったときの無限リビジョンループを防ぎます。活性化の仕方は段階的です — 1 回目は標準の二択、2 回目は「あと一回でこの選択肢が出ます」という注記付きの二択、3 回目以降で三択になります。リビジョン回数は次のステージへ移るとリセットされます。加えて Ideation・Inception フェーズでは、スコープ判定で飛ばされたステージを呼び戻す「Add Skipped Stage」という条件付きの選択肢がゲートに現れることもあります。

承認ゲートの一連の流れは、監査ログにも克明に残ります。ステージの作業が完了すると、まず要約と選択肢が(新しいタイムスタンプ付きで)監査シャードに追記され、その上で承認の問いが提示されます。あなたが Approve を選べば「ユーザーが承認した」が、Request Changes なら「変更を要求した」が、Accept as-is なら「現状のまま受け入れた」が、それぞれ記録されます。承認が済むと状態ファイルの当該ステージが完了として更新され、`Progress: 13/32 overall` のような進捗行が表示されて次のステージへ進みます。この「提示の前に選択肢を記録し、応答の後に結果を記録する」二段構えの監査が、後から「誰が・いつ・何を承認したか」を完全に再構成できるトレーサビリティを生みます。ゲートは単なる確認ダイアログではなく、人間の意思決定を記録として残すための装置なのです。だからこそ、重要な決定が記録されないまま進むことはなく、あなたはいつでも過去の判断とその理由を辿れます。

3-Strike 離脱の背後にある考え方も、押さえておく価値があります。変更要求が延々と続くとき、多くの場合は「成果物の細部」ではなく「そもそもの方針」に問題があります。Accept as-is は、完璧主義による停滞を避けつつ、現行版をアーカイブして先へ進む逃げ道を用意します。とはいえ、これは最後の手段です。1 回目・2 回目は標準の二択のままで、2 回目には「あと一回で受け入れオプションが出る」という予告が添えられます。この段階的な設計は、「安易に妥協させない」ことと「無限ループに陥らせない」ことのバランスを取っています。承認ゲートが人間の関与(HUMAN_TURN)を要求する仕組みと合わせて考えると、AI-DLC のゲート設計は「人間に、意味のある判断を、意味のあるタイミングで求める」ことに徹していると分かります。ゲートを軽い確認で済ませず、かといって人間を疲弊させもしない — この繊細な設計が、人間の監督を実効的なものにしているのです。

## 人間が決めAIが実行する

AI-DLC の思想を一言で表すのが **「User decides, AI executes(人間が決め、AI が実行する)」** です。これは方法論の第一のテネットであり、重要な決定はすべて承認ゲートを通り、人間がレビュー・修正・却下できることを意味します。承認ゲートは飾りではなく、この原則を仕組みとして強制する装置なのです。歓迎メッセージも同じ言葉で始まります — 「あなたが決め、AI が実行する。あらゆる重要な決定は承認ゲートを通る」。

これと対をなすのが **「No emergent behavior(創発的な挙動をさせない)」** という原則です。エージェントは定められたプロトコルに従い、承認メニュー・完了メッセージ・状態遷移はすべて標準化されています。その場の思いつきで三択メニューを勝手に作ったり、独自のナビゲーションを発明したりはしません。振る舞いが予測可能だからこそ、人間はゲートで安心して判断を下せます。実際 v1 の時代から、構築フェーズでは「標準化された 2 択の完了メッセージを使い、創発的な 3 択メニューを作らない」ことが明記されていました。

さらに **「Questions before assumptions(仮定より先に質問を)」** が効いています。迷ったら尋ねる — 中途半端な回答は粗い設計につながるからです。加えて **「Contradiction detection(矛盾の検出)」** により、スコープ・リスク・技術の食い違いがないか回答が相互に突き合わされます。これらの原則は互いに補強し合い、「人間の判断を中心に据えたループ」を形づくります。人間は各ステージの節目で主導権を握り、AI はその決定を忠実に実行する — 監督(oversight)と実行(execution)の役割がはっきり分かれているのが AI-DLC の要諦です。自律モードを選んだ構築フェーズでさえ、失敗は必ず人間に判断を戻します。ハンズオフは幸せな道のゲートを省くだけで、失敗時の停止は常に鳴り響くのです。

「人間が決める」を実効的にしているのが、承認が本物の人間の関与を要求する仕組みです。承認ウィジェットが人間のターンを記録しないハーネスでは、一度「approve」と短く打つだけで記録が残り、その一手があってはじめてゲートは解決へ進みます。これは「モデルが人間になりすまして承認を偽装する」ことへの、静かで確実な歯止めです。「創発をさせない」原則も、日々の体験として現れます。完了メッセージは標準化された 2 択(あるいは 3-Strike 後の 3 択)であり、エージェントが気まぐれに独自メニューを作ることはありません。振る舞いが定型で予測可能だからこそ、あなたはゲートの前で「次に何が起こるか」を正しく見通せます。裏返せば、AI-DLC における「人間の監督」とは、ただ最終承認を押すことではなく、質問の立て方・矛盾の検出・標準化されたゲート・人間ターンの要求という複数の仕組みが束になって、意思決定の主導権を人間の側に留め続けることを意味します。速さは AI が、判断は人間が — この分業を制度として支えているのが、これら一連のテネットなのです。

この監督の思想は、入門者にとってだけでなく、後段のハンズオン(茶帯)や運用・チーム(黒帯)の学習でも前提知識になります。だからこそ本単元 gates-and-human-oversight は、白帯パスと茶帯パスの両方に所属する「複数パス所属の単元」として位置づけられています。承認ゲート、Tri-Mode、3-Strike 離脱、User decides / No emergent behavior、そしてコンテキスト管理という一連の概念は、どのフェーズのどのステージにも通底する「AI-DLC の背骨」です。実際、自律モードで構築を回すときでさえ、失敗すれば必ず人間に判断が戻ることを思い出してください。AI-DLC は「人間を外す」ための道具ではなく、「人間の判断を、速さを損なわずに、確実に効かせる」ための道具です。この一点を腹に落としておけば、以降の単元でエージェントやボルト、ルールや学習ループを学ぶときも、それらがすべて「人間の監督のもとで AI が実行する」という一つの原則に奉仕していることが見えてくるはずです。

適応的な深度も、この監督と無縁ではありません。スコープや深度が「どれだけのステージを、どれだけ詳しく走らせるか」を決めるということは、裏を返せば「どこにいくつの承認ゲートが立つか」を決めるということでもあります。使い捨ての PoC なら少ないゲートで軽やかに、規制産業向けのエンタープライズなら多くのゲートで手厚く — 監督の密度そのものが案件に応じて伸縮するのです。そしてどの深度であっても、ゲートの意味論(人間が決め、AI が実行する)は変わりません。AI-DLC は「速さと監督のトレードオフ」を一律に固定するのではなく、案件ごとに最適なバランス点を選べるようにしている、と言えます。人間の監督とは、すべてを細かく確認することではなく、「重要な判断を、適切な粒度で、確実に人間が握る」ことなのだと、この設計は教えてくれます。

## コンテキスト管理とゲートでの再開

長いワークフローは、複数のセッションにまたがります。AI-DLC は進捗をすべてディスクへ永続化するので、いつでも再開・やり直し・ジャンプ・作り直しができます。中でも重要なのが **compaction(コンパクション)** への備えです。コンパクションとは、コンテキストウィンドウが埋まってきたときに、Claude Code が以前の会話コンテキストを自動で要約する処理です。

コンパクションでは「何が保持され、何が失われるか」がはっきりしています。**保持される** のは、ディスク上の成果物、`aidlc-state.md`(ステージ進捗・スコープ・プロジェクト情報)、監査シャード(意思決定と行動の全履歴)、そして `.aidlc-recovery.md`(ステージのチェックポイント)です。**失われる** のは、メモリ上の会話コンテキスト(それまでの議論)、まだファイルに書かれていない進行中の作業、そして Task ID(状態ファイルから再構築されます)やエージェントのペルソナ文脈(エージェントファイルから再読み込みされます)です。コンパクションは長いセッションでは正常な出来事であり、ディスク上の状態と成果物があるおかげで、完了済みの作業が失われることはありません。

再開時、`/aidlc` を実行すると AI-DLC は状態ファイルを読み、4 つの再開オプションを提示します — Resume from last checkpoint(最後のチェックポイントから再開)、Redo current stage(現ステージをやり直す)、Jump to stage(特定ステージへジャンプ)、Start fresh(アーカイブして作り直す)です。コンパクション直前には検証フックが隠しファイル `.aidlc-recovery.md`(復旧ブレッドクラム)を書き、最後に検証したステージ名とタイムスタンプを残します。次回起動時にこれと `aidlc-state.md` を突き合わせ、「現ステージ」がずれていれば状態破損の可能性を警告します。その場合は Redo current stage を選び、コンパクション中に進行していたステージを再実行するのが定石です。警告が出なければ、Resume from last checkpoint で通常どおり続けます。こうしてゲートと状態ファイルが、セッションをまたいでも人間の監督を途切れさせない土台になっています。

ゲート以外にも、ワークフローを操るナビゲーションがあります。`/aidlc --stage <name>` で特定ステージへ、`/aidlc --phase <name>` でフェーズの先頭へジャンプできます。前方へジャンプすると、間のステージは `[S]`(スキップ)として印が付き、下流が期待するかもしれない成果物が見つからない可能性などを、オーケストレーターが警告します。後方へジャンプすると、対象ステージは未着手にリセットされ再実行されます。加えて、ワークフローを一切変えずに現状を報告する読み取り専用のスキルもあります — `/aidlc-session-cost`(所要時間やステージの成否、メモリ項目やセンサー発火などのコスト観)、`/aidlc-replay`(その場にいなかった関係者向けの、何をなぜ決めたかの物語)、`/aidlc-outcomes-pack`(チームが再実行せずに引き継げるハンドオーバー文書 `OUTCOMES.md` を生成)です。これらは監査イベントを発さず、ステージも進めないため、途中でも安全に実行できます。状態と監査、そしてこれらの再開・ジャンプ・報告の仕組みが一体となって、長丁場のワークフローでも人間の統制が失われないようにしているのです。

4 つの再開オプションは、それぞれ「何が保たれ、何が失われるか」が明確です。**Resume from last checkpoint** は進行中または次の保留ステージから続け、すべての成果物・状態・監査を保ち、失うのは前セッションのメモリ上の会話だけです。**Redo current stage** は現ステージの成果物ディレクトリを消してチェックボックスを `[ ]` に戻し、最初からやり直します — 他の成果物は保たれ、失うのは現ステージの部分作業だけです。**Jump to stage** は特定ステージへ飛び、間のステージを `[S]` にします。**Start fresh** は、あなたの確認を取ったうえでアクティブなインテントのレコードディレクトリをアーカイブし、新しいインテントを立てます — 過去の成果物はアーカイブとして残り、失うのはアクティブなワークフロー状態だけです。状態ファイルの各ステージは 6 つのチェックボックス状態(`[ ]` 未着手、`[-]` 進行中、`[?]` 承認待ち、`[R]` 修正中、`[x]` 完了、`[S]` スキップ)で追跡され、いつ中断・再開しても現在地が一目で分かります。この「ディスク上の状態と成果物がすべて」という設計のおかげで、コンパクションやセッション終了をまたいでも、完了済みの作業と人間の判断は決して失われないのです。

コンテキスト管理を「人間の監督」の一部として捉えると、AI-DLC の設計思想がより鮮明になります。長いワークフローでは、モデルの記憶(会話コンテキスト)は当てにできません。だからこそ、監督の拠りどころを人間の記憶やモデルの記憶ではなく、ディスク上の状態ファイルと監査証跡に置くのです。人間はいつでも `/aidlc` を叩けば現在地を確認でき、必要なら特定ステージへ戻ってやり直せます。ゲートで下した判断は監査に刻まれ、コンパクションで会話が要約されても、判断そのものは残ります。つまり AI-DLC における人間の監督は、「そのセッションに張り付いていること」ではなく、「いつ戻ってきても、状態と記録から文脈を取り戻し、主導権を握り直せること」に支えられています。Tri-Mode の質問フロー、承認ゲートと 3-Strike、User decides / No emergent behavior、そしてこのコンテキスト管理 — 四つの柱が束になって、AI が実行し人間が監督するという AI-DLC の根本原則を、長丁場のワークフローでも揺るがないものにしているのです。
