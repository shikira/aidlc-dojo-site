## インテントとスペースの関係

AI-DLC を使いこなす第一歩は、**Intent(インテント)** と **Space(スペース)** という二つの器を理解することです。インテントとは「一つの作業 = ライフサイクルの一回の実行」を指します。機能追加が一つ、緊急のバグ修正が一つ、それぞれが独立したインテントです。各インテントはスペースの登録簿 `intents.json` に一行を持ち(`{uuid, slug, dirName, scope, repos, status}`)、その実行の状態・監査証跡・成果物を収める専用の「レコードディレクトリ」を持ちます。同一性を担保するのはディレクトリ名ではなく、登録簿に記録される時間順の UUIDv7 です。ディレクトリ名の `<YYMMDD>` は UTC の日付接頭辞で記録が時系列に並び、`<label>` は要求の短いケバブケース要約です。

インテントは特別なコマンドで作るものではありません。あなたが初めて作業を記述した瞬間、エンジンが自動でインテントを「誕生(auto-birth)」させます。

```
/aidlc Build a REST API for inventory management
```

まっさらなワークスペースであれば、これがインテントを生成し、`aidlc/spaces/default/intents/<YYMMDD>-<label>/` にレコードディレクトリを作り、それをアクティブにして最初のステージを開始します。ここで AI-DLC の賢さが光ります。ある機能の作業中に、無関係な別作業を記述したとしましょう。AI-DLC は「これは現在の作業とは別の、新しい仕事では?」と気づき、2 つ目のインテントを隣に立てるか確認します。勝手に増やすことは決してありません。逆に、いまのゲートへの回答や要件の修正のような「続き」であれば、アクティブなインテントの中に留めます。`active-intent` というカーソルが現在のインテントを指し、`(/aidlc intent <slug>)` のようにスラッグ名で切り替えれば、止めた場所から同じステージ・同じ状態・同じ監査証跡でそのまま再開できます。任意の数のインテントを同時に抱え、自由に行き来できます。

**Space** は「一つのチームの世界」です。`aidlc/spaces/<space>/` の下に、そのチームの `memory/`(やり方・ルール)・`knowledge/`(ドメイン知識)・`codekb/`(コード知識)・`intents/`(記録)を持ちます。既定では `default` という単一スペースが自動で用意され、ソロ開発者や単一チームはこれ以外を意識する必要はありません。複数チームが一つのプロジェクトを共有する場合にのみ、`spaces/<name>/` が同じ形で並びます。スペースを切り替えると、次に始めるインテントも、エージェントが読み込むルールや知識も、切り替え先のスペースのものになります。ここで重要な原則は「単元(ユニット)は自分の所属を知らない」ということ。作業の構造と、それを束ねる器(インテント/スペース)は、きれいに分離されているのです。

ワークスペースのレイアウトも押さえておきましょう。エンジン本体はハーネス固有の一つのディレクトリ(Claude Code なら `.claude/`、Kiro なら `.kiro/`、Codex なら `.codex/`)に収まり、あなたが日常的に開くことはありません。AI-DLC が生み出すものはすべて、中立で閲覧可能な `aidlc/` の下に、「どのハーネスで動かすか」ではなく「何に取り組んでいるか」で整理されて置かれます。あなたのコードリポジトリはワークスペースの兄弟(sibling)として並び、一つのインテントが複数のリポジトリにまたがることもできます。リポジトリの集合はインテント誕生時に自動検出され、`intents.json` の行に記録されます。git 管理の観点では、`memory/**`・`knowledge/**`・`codekb/**`・`intents.json`・各インテントの状態・監査シャード・成果物はコミットされ、チームで共有されます。一方、「いまどこにいるか」を指す 2 つのカーソル(`active-space` と `active-intent`)や、派生的な実行時スクラッチ(`runtime-graph.json` など)はユーザーごとにローカルで、gitignore されます。「カーソルとスクラッチはローカル、共有される作業はコミット」— これが原則です。

インテントの賢い扱いを、具体例で見てみましょう。ある機能の作業中に、無関係なバグ対応を記述したとします。すると AI-DLC は次のような選択肢を提示します。

```
▸ これは "inventory-api" とは別の新しい作業のようです。2つ目のインテントを始めますか?
  (1) はい — 2つ目のインテントを始める(スコープ: bugfix)
  (2) いいえ — これは inventory-api の作業の続きです
```

「はい」を選べば、AI-DLC は 2 つ目のインテント(ここでは bugfix)を誕生させて切り替え、最初の機能のインテントはそのまま保全されます — レコードディレクトリも状態も進捗も、止めた場所のまま残ります。「いいえ」を選べば、いまのメッセージはアクティブなインテントの一部として扱われます。AI-DLC が確認なしに 2 つ目を勝手に作ることはありません。ゲートへの回答や要件の修正のような明らかな「続き」であれば、そもそもこの問いは出ず、アクティブなインテントに留まります。複数チームで一つのプロジェクトを共有するなら、`(/aidlc space-create <name>)` で新しいスペースを作れます。新設のスペースはフレームワークの既定のやり方(`org.md`)と、空の `team.md` / `project.md` から始まり、他チームのプラクティスを引き継がずに自分たちの流儀を育てていきます。

## 作業単位ユニットオブワーク

**Unit of work(ユニットオブワーク)** は、解決策を独立して実装可能な単位に分解したものです。用語集の定義では「独立して実装できる、解決策の一片」であり、Inception(立案)フェーズのステージ 2.7(Units Generation)で切り出されます。大きなシステムをいきなり作ろうとするのではなく、依存関係を見極めながら、意味のある粒度のユニットへと割っていくわけです。ユニットは「何を作るか」を表す、構築フェーズの入力になります。

ユニット生成のステージは、単にユニットの一覧を出すだけではありません。`unit-of-work.md` に加えて、ユニット同士の依存関係を表す DAG(有向非巡回グラフ)を `unit-of-work-dependency.md` として生成し、さらにストーリーとの対応を示す `unit-of-work-story-map.md` も残します。依存関係の成果物の中には、各ユニットと、その `depends_on`(依存先)を列挙した必須の YAML エッジブロックが含まれます。コンパイラはこのブロックを読み取り、`runtime-graph.json` の `bolt_dag` ノードへと落とし込みます。このノードは、エッジブロックが整形式で非巡回のときにだけ現れます。そして依存が満たされる順に「バッチ」(トポロジカルな層)へと束ねます。同じバッチに入ったユニット同士は互いに依存しないため、並列に展開できます。

重要なのは、この分解の粒度が後段の並列度を左右する点です。粗く、相互依存の少ないユニットに割れば、多くのユニットが同じバッチに入って同時に走ります。逆に細かく深く連鎖する依存を持たせると、作業は多数の小さなバッチへと直列化されます。この分解自体はアーキテクト・エージェントが人間とともに行う知識判断であり、そこで書かれたトポロジーをコンパイラがバッチへと変換します。そして一つ以上のユニットが束ねられて、次に説明する **Bolt(ボルト)** を構成します。ユニットは「何を作るか」を、ボルトは「どう回すか」を担う単位、と捉えると関係が見えてきます。なお、この並列面(per-Unit のステージ群)が存在するのは、ユニット生成が走るスコープ(enterprise・feature・mvp・workshop)に限られます。

裏を返せば、増分的なスコープ(bugfix・refactor・security-patch)や poc・infra では、ユニット生成そのものが走りません。エッジブロックも `bolt_dag` も生成されず、構築は「一巡(single-pass)」で進みます。並列に散らす相手がいないからです。だからこそ、swarm による並列展開や「ハンズオフな構築」は、本質的に複数ユニットを持つグリーンフィールドのスコープの性質だと理解しておくとよいでしょう。ユニット生成のステージは、構築フェーズの並列度を決める「上流のダイヤル」なのです。粒度を粗くすれば同時実行が増え、細かくすれば直列化する — チームがどんなガイダンス(ルール)をアーキテクトに与えるかが、この分解に影響します。ユニットとストーリーの対応を残す `unit-of-work-story-map.md` は、要件→ストーリー→ユニットというトレーサビリティの鎖をつなぎ、フェーズ境界検証が「各要件が実装単位まで届いているか」を確かめる土台にもなります。

用語の関係を整理しておきましょう。「ユニット(Unit of work)」は分解の単位=何を作るか、「ボルト(Bolt)」は実行の単位=それをどう回すか、です。一つのボルトが一つのユニットに対応することも、依存でつながった小さなユニット群をまとめて一つのボルトにすることもあります。ユニットは Inception のステージ 2.7 で生まれ、ボルトは Construction でそのユニットを回す一巡として立ち上がります。ユニットが増えれば、依存関係次第で並列バッチの幅が決まり、ボルトの並びが決まります。つまり「ユニットへの良い分解」が、そのまま「レビュー可能で、必要なら並列化できるボルト列」へと直結するのです。この上流の設計判断が構築フェーズの体験を大きく左右するからこそ、ユニット生成は Inception の中でも特に重要なステージとして位置づけられています。

## 実行単位ボルトとウォーキングスケルトン

**Bolt(ボルト)** は、Construction(構築)フェーズの実行単位です。用語集の定義によれば、ボルトとは「一つのユニット(または依存でつながった小さなユニット群)に対する、ステージ 3.1〜3.5 の一巡」を指します。機能設計・NFR 要件・NFR 設計・インフラ設計・コード生成をひとまとまりで回し、レビュー可能な一枚のスライスとして仕上げます。なお、ステージ 3.6(ビルド&テスト)と 3.7(CI パイプライン)はボルトごとではなく、全ボルト完了後に一度だけ実行されます。

この形には歴史があります。かつて Construction はユニットごとにステージを一つずつ回し、各ステージのあとに承認ゲートを置いていました。3 ユニットの案件なら、テスト済みコードが 1 行出る前に 15 回もゲートが挟まり、「子守のようだ」と不評でした。そこで一度は全ユニットの質問・設計・コード生成をまとめ、最後に一回だけレビューする方式に振れましたが、今度は 15 ユニットで 15,000 行が一気に出て、一度のレビューでは検証しきれませんでした。現在の「ボルト単位」は、その振り子の中間解です。2.8 のデリバリー計画があらかじめ決めたボルトの並びに沿って、レビュー可能な大きさのスライスを刻んでいきます。

最初のボルトは特別で、**Walking Skeleton(ウォーキングスケルトン)** と呼ばれます。これはアーキテクチャを実証する「端から端まで動く最小の縦断スライス」で、あらゆる統合点を一度は貫きます。ウォーキングスケルトンは常にゲート付きで対話的に進み、あなたが全体の形を確認してから残りが動きます。そして承認の直後、**ladder prompt(はしごの問い)** が一度だけ発火します — 「残りのボルトはどう回す? 自律的に続けるか、ボルトごとにゲートを設けるか」。あなたの答えは `Construction Autonomy Mode` として状態に記録され、以降のすべてのボルトを支配し、セッションを再開しても尊重されます。依存が満たされ互いに独立したボルトは **parallel batch(並列バッチ)** として同時に走り、コンダクターが一度のターンで複数の `Task` 呼び出しを発行します。バッチ末尾の単一ゲートが全ボルトをまとめて面倒を見ます。ただし失敗は必ずワークフローを止め、自律モードであっても retry(再実行)/ skip(スキップ)/ abort(中止)を人間に尋ねます。これは自律モードで唯一割り込む場面です。

ボルトの並列展開を担うのが **swarm(スウォーム)** の仕組みです。多数のユニットへ作業を扇状に広げますが、フレームワークはワーカーの「収束した」という自己申告を鵜呑みにしません。権威ある信号は、プロジェクト自身のチェックコマンド(`npm test` などの、ビルドやテストのスクリプト)です。終了コード 0 が本当に収束した証で、それ以外は「まだ」を意味します。さらに、収束の基準を定めるテストファイルが勝手に弱められていないか、フォーク元との anti-tamper 比較も行われます。並列駆動の選択は運用者のノブ、収束の判定はツール、そして「どこまで自律で回すか」の姿勢(posture)はチームがルールとして書く — この「決定的なものはツールへ、知識はエージェントへ、判断は人間へ」という線引きが、構築フェーズには一本きれいに引かれています。ウォーキングスケルトンとはしごの問い、自律モード、失敗時の halt-and-ask は、いずれもこの線引きの表れなのです。

## スモールモブと11のエージェント

最後の基本概念が **Small Mob, Broad Agents(小さなモブ、広いエージェント)** という設計思想です。AI-DLC は、細分化された多数の専門家ではなく、複数のステージ・フェーズをまたいで働く **11 の広く有能なエージェント** を用います。3〜5 人のモブが一機能を丸ごと担う人間チームの姿を、そのままソフトウェアに写したものです。「なぜ 30 ではなく 11 なのか」— それは、各エージェントが一つの領域を多くのタスクにわたって受け持ち、エージェントの境界(=情報の喪失点)を減らすためです。

11 のエージェントは、プロダクト・デザイン・デリバリー・アーキテクト・AWS プラットフォーム・コンプライアンス・DevSecOps・開発者・品質・パイプライン&デプロイ・運用の各ペルソナです。コンダクターがステージに応じてこれらを有効化します。アーキテクト・エージェントは 3 フェーズ・9 ステージにまたがる最も広い関与を持ちます。コンプライアンスや DevSecOps のように、自身が主導するステージを持たず、他のエージェントが率いるステージに「サポート役」として加わるものもあります。こうすることで、レビュー専用エージェントを乱立させずに横断的関心事をカバーできます。運用エージェントは、運用上の知見をプロダクト・エージェントへ還流させ、ライフサイクルのループを閉じる役割も担います。

この思想の背後にはモブプログラミングの実践があります。モブプログラミングとは「チーム全員が、同じものに、同じ時に、同じ場所で、同じコンピューターで取り組む」こと。ドライバー(キーボードを打つ人)とナビゲーター(考え、方向づける人)が役割を回しながら進めます。狙いは、リアルタイムの知識共有、属人化(バス係数)の低減、そしてレビューを開発と同時に行うことによる品質向上です。推奨されるドライバー交代の間隔は 10〜15 分で、全員が例外なくローテーションに参加します。AI-DLC のエージェント設計は、この「少人数で広く、引き継ぎを減らす」精神を、11 という数に凝縮しているのです。そして委譲は常にコンダクターが行い、エージェント同士が直接呼び合うことはありません — この一点も、予測可能で監査可能な協働を保つための重要な設計です。

エージェント同士の情報の流れには、はっきりした形があります。プロダクト・エージェントが要件・ストーリー・スコープをアーキテクト・エージェントへ渡し、アーキテクトが仕様を開発者エージェントへ、開発者がコードを品質エージェントへ、品質がテスト結果とバグ報告を開発者へ返す。AWS プラットフォーム・エージェントが用意したインフラをパイプライン&デプロイ・エージェントがデプロイし、運用エージェントへ渡す。そして運用エージェントが運用上の知見をプロダクト・エージェントへ還流させ、ライフサイクルのループを閉じます。ツールの観点では、11 のエージェントはセッションの全ツールセットを継承し、唯一の共通制限は「Task を使えるのはコンダクターだけ」という点です。Bash を実際に使うのは AWS プラットフォーム・DevSecOps・開発者・品質・パイプライン&デプロイ・運用の各エージェント、WebSearch を使うのはプロダクト・デザイン・コンプライアンスの各エージェント、という具合に「どのツールを行使することが期待されるか」が役割ごとに整理されています。少数の広いエージェントが、明確な情報の流れと役割分担のもとで協働する — これが AI-DLC の「小さなモブ」の実像です。

これら四つの基本概念は、互いに噛み合って一つの流れを作ります。**スペース**(チームの世界)の中で **インテント**(一つの作業)が生まれ、Inception で **ユニット**(実装単位)へと分解され、Construction で **ボルト**(実行単位)として回され、その全体を **小さなモブ**(11 のエージェント)が担う。ユニットが「何を」を、ボルトが「どう」を、モブが「誰が」を受け持ち、インテントとスペースが「どこに記録するか」を定める、と整理できます。各エージェントはステージ開始時に、方法論知識(フレームワーク同梱)とチーム知識(スペースの `knowledge/`)を読み込み、承認ゲートを経て成果物を残していきます。用語をひとつずつ押さえておけば、以降のハンズオン(茶帯)や運用(黒帯)の単元で、これらの概念が具体的なコマンドや成果物としてどう現れるかが、すっと頭に入るはずです。

最後に、これらの概念がどのバージョンに属するかを確認しておきましょう。Intent・Unit・Bolt・Small Mob という四つの基本概念は、v2 で整理・実体化されたものです。特に Unit や Bolt は v2 の構築フェーズを駆動する中心概念であり、v1 の「ユニットごとのループ」から「ボルト単位 + 並列バッチ」への進化を通じて、その意味がはっきりしました。本サイトは v2 を主軸に据えているため、これらの用語も v2 の定義に沿って説明しています。とはいえ、モブ(Small Mob, Broad Agents)の思想は AI-DLC 全体を貫く設計哲学であり、バージョンを問わず通底します。基本概念を押さえることは、単に用語を覚えることではなく、「AI-DLC がどんな粒度で作業を捉え、誰がそれを担い、どこに記録するか」という世界観を身につけることです。この土台の上に、次の単元で学ぶ 5 フェーズの全体像や、承認ゲートと人間の監督が積み上がっていきます。
