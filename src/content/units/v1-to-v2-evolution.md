AI-DLC は v1(1.x)系列から v2(2.x)系列へと大きく形を変えました。このサイトの主軸は現行の v2 ですが、v1 は正式リリース済み(1.0.1)のレガシーとして、対比と移行のために知っておく価値があります。この単元では、v1 系列の現在地、v2 の到達点、そして両者をつなぐ進化と移行の道筋を、公式ドキュメントに忠実に追います。専用の移行ガイドは存在せず、移行は各版 CHANGELOG の「Upgrade」注記で表現される、という点が黒帯として押さえどころです。v1 を貶めるためではなく、「v2 が何を得たのか」を対比で理解し、旧構成からの移行を安全に進めるための単元だと捉えてください。v1 は 3 フェーズ・2 層ルール・extensions opt-in という骨格を持ち、v2 は 5 フェーズ・5 層ルール・学習ループ・ワークスペースモデルへと拡張した — その差分と、差分を埋める移行の実務(CHANGELOG の Upgrade 注記)を順に見ていきます。

## v1(1.x)系列の現在地

v1 の `aidlc-rules/VERSION` は `1.0.1` を示します。これは正式リリース済みの安定版で、v1 の README も 2.0 を「Preview」として案内しつつ、安定を求める利用には v1 系列が現役であることを前提にしています。v1 の中核は **三フェーズのアダプティブワークフロー** で、`core-workflow.md` は INCEPTION → CONSTRUCTION → OPERATIONS の 3 段で構成されます。その基本理念は「**ワークフローが作業に適応する(その逆ではない)**」— AI モデルがユーザーの意図と明確さ・既存コードベースの状態・変更の複雑さとスコープ・リスクと影響を評価し、どのステージが必要かを判断します。この「必要なステージだけ走らせる」思想は v2 のスコープ/深度にそのまま受け継がれます。

v1 の INCEPTION は Workspace Detection(常に実行)・Reverse Engineering(ブラウンフィールドのみ条件付き)・Requirements Analysis(常に実行・アダプティブ深度)・User Stories(条件付き)・Workflow Planning(常に実行)・Application Design(条件付き)・Units Generation(条件付き)を含みます。CONSTRUCTION は Unit ごとのループ(Functional Design・NFR Requirements・NFR Design・Infrastructure Design を各 Unit で回し、Code Generation は常に実行)と、最後の Build and Test を回します。そして OPERATIONS は **PLACEHOLDER(プレースホルダ)** で、実体はまだ無く、「すべてのビルドとテスト活動は CONSTRUCTION フェーズで扱われる」と明記されています。Build and Test の後に「Operations ステージへ進む準備はできましたか?」と明示的な承認を求める作りで、Operations 自体の中身が空だったことが、v2 との最大の差の一つです。

v1 のルール構造は **2 層** です。`aws-aidlc-rules/core-workflow.md` が全体を統べ(「このワークフローは他の組み込みワークフローすべてに優先する」と冒頭で宣言)、詳細は `aws-aidlc-rule-details/`(`common/`・`inception/`・`construction/` など)の rule-details に置かれます。ルール詳細のパスは、AI 支援セットアップ・Cursor/Cline/Claude Code/Codex・Kiro・Amazon Q など複数の IDE / セットアップ方法に対応するよう、`.aidlc/`・`.kiro/`・`.amazonq/` などを順に探す複線解決になっています。加えて `extensions/` の **opt-in 方式** があり、起動時には各拡張の `*.opt-in.md`(軽量なオプトインプロンプト)だけを読み、ユーザーがオプトインしたときに初めて対応する完全なルールファイルを読み込む — コンテキストを節約する仕組みです。質問は A/B/C/D/E の多肢選択と `[Answer]:` タグ、Mermaid や ASCII 図の検証を伴うコンテンツ検証、カスタムウェルカムメッセージなど、v1 は「ルール文書としての厳密な MANDATORY 指示の束」という性格が強いのが特徴です。

v1 の各ステージの条件付き実行も、v2 のスコープ思想の原型です。Reverse Engineering はブラウンフィールド(既存コードがある)ときだけ走り、User Stories・Application Design・Units Generation は複雑さに応じて条件付きで実行されます。Requirements Analysis はアダプティブ深度で、単純な要求なら軽く、複雑なら深く分析します。拡張(extensions)は opt-in で読み込んだものだけが効きますが、いったん有効になると **ハードな制約** で、適用可能な拡張ルールへの非準拠は **blocking finding**(ブロッキング指摘)となり、解決するまでステージ完了を提示しない、という強制力を持ちます。各ステージ完了時には拡張ルールの準拠状況(compliant / non-compliant / N/A、N/A には根拠)のサマリを添えます。成果物は `aidlc-docs/` 配下に inception / construction / operations のフェーズ別ディレクトリで整理され、プロンプトと判断は `audit.md` に記録される — この「バージョン付き成果物 + 監査ログ」という発想自体は v2 にも受け継がれますが、v2 では per-intent のレコードディレクトリと per-clone シャードへと構造が進化します。要するに v1 は、思想(アダプティブ・承認・トレース)は既に持ちつつ、実装は単一のフラットな `aidlc-docs/` と 2 層ルールに留まっていた段だと理解できます。v1 が正式リリース済み(1.0.1)であることは重要で、これは「試作品」ではなく実運用された安定版です。だからこそ v1 を知りたい層に向けて、貶めるのではなく選択肢として案内する — 現行は v2 だが、v1 の 3 フェーズ・2 層ルール・extensions opt-in という骨格を理解しておくと、既存の v1 プロジェクトの読み解きや、v2 との差分の把握が容易になります。v1 のルール文書が「MANDATORY」を多用する強い指示体であったこと、Operations が空だったこと、拡張がオプトインで有効化されたときだけ効いたこと — これらは v2 の学習ループ・実体化した Operation・5 層ルールと対比すると、進化の方向がくっきり見えてきます。

## v2(2.x)の到達点

v2 は AI-DLC 方法論のネイティブ実装で、「**one core, many harnesses(ひとつのコア、多くのハーネス)**」を掲げます。方法論はハーネス中立な `core/` に一度だけ宿り、各ハーネス(Claude Code・Kiro IDE・Kiro CLI・Codex CLI、そして移植すれば任意の能力あるハーネス)は「そのハーネスでどう見せるか」を決める薄い表層を足すだけ。だから方法論の編集は 1 か所で済み、各ハーネスの配布物はそこから **生成** され、どのハーネスも特別扱いされません。README は v2 を **GA Preview(活発に開発中)** と位置づけ、インターフェース・ステージ定義・エージェント構成・インストールモデルがまだ進化中で版間に破壊的変更が入りうると警告し、**本番利用は安定した `main` ブランチに留まる** よう案内します。

到達点を数字で見ると、v2 は **5 フェーズ・32 ステージ・11 エージェント・9 スコープ(auto-detect 付き)・3 深度・3 テスト戦略・68 種の監査イベント** を持ちます。3 フェーズだった v1 に対し、v2 は Initialization と Ideation を前段に加えて 5 フェーズになり、そして v1 で PLACEHOLDER だった Operation が 7 ステージの実体(4.1〜4.7)になりました。加えて Bolt と swarm による Construction、あらゆる判断を通す承認ゲート、人間の訂正を恒久ルールに変える **学習ループ**、二層の Knowledge、per-clone シャードの監査証跡 — これらが v2 で揃いました。v1 の「MANDATORY 指示の束」から、v2 の「決定的エンジン + コンダクター + データで形づくる構成」へと、実装の性格そのものが変わっています。決定的なエンジン — 状態機械・監査ログ・グラフ・swarm レフェリー・学習ゲート — は全ハーネスの配布物で **バイト同一** で、違うのは shell(ゲートの描画・サブエージェントの起動・フックの結線・起動方法)だけ、という点も v1 には無かった v2 の設計上の到達です。

v2 の版の歩みは CHANGELOG に刻まれています。**2.0.0**(2026-06-18)で v2 ラインに rebase・統合し、reviewer 機構と Kiro IDE ハーネス(13 エージェント構成)を取り込みました。**2.1.0**(2026-06-24)は単一フラットな `aidlc-docs/` から **per-intent ワークスペース** レイアウト(spaces・複数 intent 共存・マルチリポ・space レベルの knowledge/code-KB・アクティブ space に追従する memory)へ移し、`/aidlc --init` を廃止して初回起動での intent auto-birth に変えました。**2.1.7・2.1.8** は incremental スコープの dangling-consume や reviewer をハーネス横断で整え、**2.2.0**(2026-07-04・このサイトの照合基準・CHANGELOG=2.2.0)は Adaptive Workflows — タスクにセレモニーを合わせる composer エージェント(`/aidlc compose`)— を追加しました。「現行 = v2、v1 は正式リリース済みのレガシーで対比・移行のために扱う」というのが、この教材全体の立ち位置です。

v2 が v1 に対して足したものを、利用者視点でもう少し具体的に挙げると次のとおりです。**9 つのアダプティブスコープ**(enterprise から workshop)がフリーテキストの意図から auto-detect され、2.2.0 の **adaptive composer**(`/aidlc compose`)はタスク・スキャンレポート・走行中ワークフローから、あつらえのステージ計画を提案します。**3 つの深度レベル**(Minimal/Standard/Comprehensive)と、それとは独立した **3 つのテスト戦略レベル** が、成果物の詳細度とテスト網羅を別々に制御します。CLI ユーティリティで任意のステージ/フェーズへジャンプでき、状態確認やスコープ/深度/テスト戦略の途中変更も可能です。**セッション resume** は checkpoint からの継続・redo・stage ジャンプ・fresh start を提供し、**二層の Knowledge**(方法論はフレームワーク同梱、チーム知識はユーザー管理)と **ルール + 学習ループ**(人間の訂正が恒久ルールになる)が、v1 には無かった「使うほど賢くなる」性質をもたらします。監査は 68 種のイベントで構造化ログとして残り、企業のトレーサビリティ要件に応えます。エージェント構成も v1 の暗黙的な役割から、11 のドメインエキスパートペルソナ(2.0.0 では reviewer を含む 13 構成)へと明確化されました。要するに v2 は、v1 の思想を保ちながら、それを回すためのエンジン・スコープ体系・知識/学習の仕組み・ハーネス中立なコアを実装として整えた到達点だと言えます。数字で対比すれば、フェーズは 3 → 5、ルール層は 2 → 5(org → team → project → phase → stage)、そして空だった Operations が 7 ステージの実体になった、という三点が v1 から v2 への最も分かりやすい進化の指標です。この三点を押さえておけば、v1 のドキュメントに触れたときも、v2 との差分をより素早く正確に見取れるはずです。

## 進化の経緯と移行(CHANGELOG の Upgrade 注記)

v1 から v2 への移行に、専用の移行ガイド文書は **ありません**。代わりに、移行の実務は各版 CHANGELOG の「Upgrade」注記に集約されています。そしてその注記は一貫して同じ動作を指します — **「re-copy your `dist/<harness>/` shell into the project(プロジェクトへ `dist/<harness>/` の shell を再コピーする)」**。v2 は方法論を `core/` に置き各ハーネスの配布物 `dist/<harness>/` を生成する設計なので、新しい版を取り込むとは「生成済みの配布物を自分のプロジェクトへ入れ直す」ことに他なりません。2.2.0・2.1.8・2.1.7・0.7.x… と、ほぼ全ての版の Upgrade 注記がこの再コピーを指示しており、これが v2 における「アップグレード」の標準動作です。

大きな移行は 2.1.0 に集中しています。旧来のフラットな `aidlc-docs/` レイアウトに残るプロジェクトは、初回実行時に **自動かつクラッシュ安全に移行** されます(コピー → per-intent レコードへの原子的リネーム → 追跡されていたフラットツリーを git-rm、`.migrated` マーカーで冪等)。監査証跡は単一の `audit.md` から **per-clone シャード**(`intents/<id>/audit/<host>-<clone>.md`)へ変わり、並行クローンがトレイルをマージ衝突させないようになりました(読み手は glob してタイムスタンプでマージ)。確定した学びは新しい `aidlc/spaces/<space>/memory/` に **プラクティス** として着地し(旧 `*-learnings.md` は廃止)、チームのドメイン知識は space レベルの `aidlc/knowledge/` へ自動的に移されます。そして `/aidlc --init` は退役し、出荷される shell がワークスペースを事前 scaffold し、初回 `/aidlc` で最初の intent が auto-birth します。

CHANGELOG を版ごとに読むと、移行時に注意すべき breaking change が具体的に分かります。**2.0.0** は v2 ラインへ rebase して reviewer 機構と Kiro IDE ハーネスを統合し、13 エージェント構成と 2 つの Kiro 配布を持ち込みました。**2.1.0** のワークスペース化は既に述べたとおり、フラット `aidlc-docs/` の自動移行・監査の per-clone シャード化・学びのプラクティス化・知識の space レベル化をまとめて行い、`--init` を廃止しました。加えて、人間不在のまま自動運転のモデルが承認を捏造できないよう、承認と質問のゲートは **HUMAN_TURN** の記録を要求するようになり(前回のゲート解決以降に人間のターンが無ければ approve を拒否)、合成 CI では `AIDLC_SKIP_HUMAN_PRESENCE_GUARD=1` で回避します。**`--test-run` フラグと Test Run Mode は完全撤去** され(自分の未完成の仕事を自動承認できてしまう穴を塞ぐため)、これに依存していた CI やスクリプトは `AIDLC_SKIP_ARTIFACT_GUARD=1` などへの置き換えが必要です。ステージを完了させる各サブコマンドは、宣言された `produces[]` 成果物がディスクに無いとステージを `[x]` にしない artifact guard も入りました。これらは「エージェントがゴム印を押すように空の完了を積む」罠を塞ぐ、v2 で強化された統制です。

移行を考えるうえでの心構えは 2 つです。第一に、v1 と v2 は思想の連続体だという点 — アダプティブに必要なステージだけ走らせる、ユーザーが決めて AI が実行する、承認ゲートで統制する、といったテネットは両系列を貫きます。だから v1 でワークフローの流れを掴んだ人は、v2 でも同じ地図が使えます。第二に、**破壊的変更が版間で入りうる** という点で、v2 README も「依存するものは既知の良好な版にピン留めし、生成された出力は行動前に必ずレビューせよ」と繰り返し警告します。前段で挙げた `--test-run` 撤去や HUMAN_TURN ガードのように、CI やスクリプトが影響を受ける breaking change は現に起きています。v1 で `/aidlc --init` に慣れた人は、v2 では init が廃止され初回 `/aidlc` で intent が auto-birth する、といった具体的な差分を、実際に `dist/<harness>/` を再コピーして手を動かしながら確かめるのが確実です。専用ガイドが無いぶん、CHANGELOG の各 Upgrade 注記と README がそのまま移行の一次情報になる — 移行のたびに CHANGELOG のその版のエントリを読み、Upgrade 注記に従って `dist/<harness>/` を入れ直し、breaking change があれば自動化を直す、という運用が v2 時代の標準的なアップグレード手順になる、というのがこの節の要点です。

実務的な移行チェックとしては、まず対象ハーネスの `dist/<harness>/` を再コピーする(Claude なら `dist/claude/.claude/`、Kiro なら `dist/kiro/.kiro/` と `AGENTS.md`、Codex なら `dist/codex/`)、次にフラット `aidlc-docs/` のプロジェクトは初回 `/aidlc` の自動移行に任せる(手動移行は不要・冪等)、そして CI が古いフラグ(`--test-run` など)や `--init` に依存していないか確認する、という 3 点が軸になります。v2 は pre-1.0 の GA Preview として活発に動いているため、Upgrade 注記の指示が版ごとに増えることもあります — だからこそ「専用ガイドを探す」のではなく「その版の CHANGELOG エントリを一次情報として読む」習慣が、v1 から v2 への、そして v2 内での安全な移行を支えます。v1 を選択肢として残しつつ現行の v2 を主軸に据える、という本教材の立場も、この「版ごとに一次情報を確認する」姿勢の上に成り立っています。
