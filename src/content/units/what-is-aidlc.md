## AI-DLCの定義と起源

AI-DLC(AI-Driven Development Life Cycle)は、AI 支援によるソフトウェア開発を、再現可能でトレース可能なフェーズへと構造化するための方法論です。AWS が提唱した [AI-DLC メソドロジー](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/) に由来し、公式リポジトリ `awslabs/aidlc-workflows` は、この方法論を「ハーネス中立なコア(core)」からネイティブに実装しています。つまり方法論の本体を一つのソースに一度だけ書き、そこから複数のハーネス向けの配布物を生成します。結果として、あなたが普段使っている CLI ツール — 現在は Claude Code、Kiro CLI、Kiro IDE、Codex CLI、そしてあなたが移植する任意の対応ハーネス — の中で、そのまま動きます。

利用者は、たった一つのコマンドから始めます。

```
/aidlc Build a REST API for inventory management
```

これだけで AI-DLC は、インテント(意図)の把握から、要件定義・設計・実装・テスト・デプロイまでを、構造化されたワークフローとして順に案内します。しかも各意思決定ポイントでは常に人間が主導権を握ります。Codex CLI では起動子が `$aidlc` になるといった違いはありますが、方法論そのもの — フェーズ、ステージ、エージェント、ゲート — はどのハーネスでも同一です。差が出るのは「見え方」だけであり、状態機械・監査ログ・並列エージェントを調停するレフェリーといった決定的なエンジンは、どのハーネスでもバイト単位で同じものが動きます。

なぜ、わざわざ構造を持ち込むのでしょうか。公式リポジトリの README は率直です。「行き当たりばったりの AI コーディングは、プロジェクトが本格化するまでは通用する」。本格化すると、プロンプト間でコンテキストがずれ、決定の理由が記録されず、モデルが頼んでもいないことを静かにやってしまう。AI-DLC は作業のまわりに枠組みを置くことでこれに対処します。各ステージには明確な担当がいて、あらゆる決定は次が始まる前に承認ゲートを通過し、この実装はあなたの修正から学んだことを記録して、同じ間違いを繰り返さないようにします。生成 AI は誤りを犯しうるものなので、出力とコストは常に人間がレビューすべきだ、と AWS も明記しています。

AI-DLC は「方法論」であって、それ自体はコードではありません。AWS が定義した構造化・ゲート付きの AI 駆動開発アプローチであり、`awslabs/aidlc-workflows` リポジトリはそれを「スキル・エージェント・フック・ツール」としてハーネス中立な `core/` からレンダリングした、ネイティブなマルチハーネス実装です。方法論が「何を(what)」を定め、各ハーネス配布物が「あるランタイム上でどう実現するか(how)」を担う、という関係です。より深く知りたい場合は、AWS の [ブログ記事](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/) や、それが参照するメソッド定義ペーパー(Method Definition Paper)に当たるとよいでしょう。なお公式リポジトリの v2 系列は執筆時点で GA プレビュー(2.x)であり、インターフェイスやステージ定義、エージェント構成、インストール方式はなお進化の途上にあります。だからこそ、依存する箇所は既知の良好なバージョンに固定し、生成された出力は必ずレビューしてから使うことが推奨されます。

なお本サイトの学習は **現行の v2(2.x 系列)を主軸** に構成しています。v1(1.0.1)は正式リリース済みの系列ですが、いまはレガシー/対比・移行のための位置づけとして扱います。v1 を貶めるのではなく、v1 を知りたい層のための選択肢として案内する、という立場です。既定の学習体験は v2 を前提に書き、v1 はフェーズの対比や移行の文脈で言及します。

## 5つのフェーズによるライフサイクル

AI-DLC のライフサイクルは、5 つのフェーズと、そこに属する 32 のステージで構成されます。フェーズは順に次の 5 つです。

- **Initialization(初期化)** — 状態ファイルやディレクトリの雛形づくり、ワークスペースのスキャン、ルーティング。ワークフローの準備が整った状態(configured workspace)を作ります。3 つのステージからなり、承認ゲートなしで自動的に走ります。
- **Ideation(構想)** — インテント、市場調査、実現可能性、スコープ定義、チーム編成を通じて、着手する取り組みそのものを検証します。成果は承認済みの「イニシアチブ・ブリーフ」です。7 ステージからなります。
- **Inception(立案)** — 要件、ユーザーストーリー、設計、アーキテクチャ、ユニット、デリバリー計画へと精緻化し、詳細な実行計画をまとめます。8 ステージからなり、「WHAT / WHY(何を・なぜ)」を扱います。
- **Construction(構築)** — 機能設計、NFR(非機能要件)、インフラ、コード、テスト、CI を経て、動作しテスト済みのコードを作ります。7 ステージからなり、「HOW(どう作るか)」を扱います。
- **Operation(運用)** — パイプライン、環境、可観測性、インシデント対応、フィードバックを整え、監視付きの本番システムへと仕上げます。7 ステージからなります。

フェーズは順番に実行されます。そしてフェーズの境界(Initialization → Ideation を除く)では **検証ゲート(verification gate)** が自動でトレーサビリティを点検します。要件がストーリーに、ストーリーがアーキテクチャに繋がっているか、孤立した成果物や欠落した参照がないかを、下流のステージが積み上がる前に確かめるのです。検証に失敗すれば、コンダクターは問題を報告し、先に進むか戻って直すかをあなたに尋ねます。

AI-DLC の主要な数字を押さえておきましょう。フェーズは 5、ステージは 32、エージェントは 11、スコープは 9 種類(enterprise から workshop まで、フリーテキストからの自動判定つき)、深度レベルは 3(Minimal / Standard / Comprehensive)、テスト戦略レベルは 3、そして監査イベントの種類は 68 に及びます。これらの数字は、方法論全体の骨格を端的に示しています。各フェーズが 3〜8 のステージを持ち(Initialization 3、Ideation 7、Inception 8、Construction 7、Operation 7)、それぞれに担当エージェントと定められた入出力があります。

ここで押さえておきたいのが「スコープ」「深度」「テスト戦略」という三つの調整軸です。**スコープ** は 9 種類(enterprise、feature、mvp、poc、bugfix、refactor、infra、security-patch、workshop)あり、どのステージを実行しどれだけの深さで扱うかを決めます。フリーテキストの意図から自動判定されるほか、明示的に選ぶこともできます。**深度** は成果物の詳しさを 3 段階で制御し、**テスト戦略** はテストの量と種類を深度とは独立に 3 段階で制御します。これらは承認ゲートでいつでも上書きできます。単純なバグ修正なら多くのステージが省かれ、規制産業向けのエンタープライズ案件なら全 32 ステージがフルの深度で走る — 同じ方法論が、この振れ幅を一つのエンジンで吸収するのです。ユーザーガイドも、新規ユーザーは Getting Started から、常用ユーザーは CLI コマンドやスコープ・深度の章から、チームリードは知識とルール・学習ループの章から、という具合に読者に応じた導線を用意しています。

## EngineとConductorのオーケストレーションループ

AI-DLC の中核は、驚くほど素朴なループです。決定的な **Engine(エンジン)** が「次に何をするか」を決め、**Conductor(コンダクター)** — すなわち `/aidlc` セッション本体(`SKILL.md`)— がそれを実行し、終わったら再びエンジンに次の一手を尋ねます。このループの中で、フレームワークは次のことを行います。

1. **ステージファイルを読む** — 5 フェーズにまたがる 32 のステージ定義。各ステージが入力・手順・出力・担当エージェントを規定します。
2. **エージェントのペルソナを読み込む** — アーキテクト、開発者、プロダクトマネージャーなど、専門的な知識を持つ視点を有効化します。
3. **状態と監査を管理する** — 進捗を `aidlc-state.md` で追い、あらゆる決定をインテントの監査ログのシャードに記録して、意図から本番までのトレーサビリティを確保します。
4. **サブエージェントへ委譲する** — 集中的で自律的な作業を要するステージ(リバースエンジニアリング、コード生成)では、サブプロセスを起こします。
5. **承認ゲートを提示する** — 各ステージのあと、あなたが確認し承認してから、ワークフローは前へ進みます。

役割分担は明快です。エンジンは **ルーティング**(どのステージが次か、どのスコープか、いつ止まるか、ジャンプや再開、ゲートの状態)を担い、コンダクターは **実行の質**(ステージをうまく回す、良い問いを立てる、決定をあなたに提示する)を担います。技術的には、エンジンは `aidlc-orchestrate.ts`(`next` / `report` サブコマンド)という決定的なツールで、コンダクターが従うべき型付きの「Directive(指示)」— 例えば `run-stage`、`ask`、`print`、`done` — を発します。コンダクターはその指示に従って動き、結果を報告し、また次を尋ねる。この往復がループの実体です。

ほとんどのステージは **インライン** で走ります。コンダクターがエージェントの視点を借りて、会話の中で直接あなたと作業します。一方、リバースエンジニアリング(2.1)とコード生成(3.5)の 2 つだけは **サブエージェント** として、バックグラウンドのサブプロセスに委ねられ、あなたと対話することなく自律的に実行され、完了後に結果が提示されます。いずれの場合も、委譲を行うのは常にコンダクターであり、エージェント同士が直接呼び合うことはありません。この「決定はエンジン、実行はコンダクター」という分離が、予測可能で監査可能なワークフローを支えています。

このループの周辺には、いくつかの支えがあります。まず **フック(hook)**。ハーネスがイベントに応じて自動実行する TypeScript スクリプト群で、監査ログの記録、センサーの発火、状態の検証、セッションのライフサイクル管理などを担います。いずれも「アクティブなワークフローがなければ何もしない」よう自己ゲートされています。次に **監査証跡**。インテントのレコードディレクトリの `audit/` に、追記専用のイベントログとして書かれます。68 種類のイベントが ISO タイムスタンプ付きで並び、意図から本番までを追跡できます。さらにフレームワークは関心事を三つの「プレーン」に分けて捉えます — ステージ定義やルール・センサーからなる制御プレーン、実際のステージ実行やボルト・監査テレメトリからなるデータプレーン、そして `--doctor` や監査クエリからなる管理プレーンです。この分離が、エンジンとコンダクターの素朴なループを、堅牢で観測可能な仕組みへと押し上げています。

ループの進行は、あなたの手元にも見えます。ターミナルのステータスラインには、現在のフェーズ、ステージの表示名、フェーズ進捗バー、フェーズ内の進捗比、担当エージェント、そして残りコンテキスト量(`ctx:N%`)が表示されます。たとえば `[AIDLC] IDEATION > Intent Capture [▓▓▓▓▓░░░░░] 4/7 -- product` のように、いま何をしているかが一目で分かります。承認のたびに `Progress: 4/32 overall | 1/7 IDEATION stages complete. Next: Market Research` といった進捗行も表示されます。技術的には、エンジンが発する指示(Directive)には `run-stage`(ステージを実行)・`ask`(問う)・`print`(表示)・`done`(完了)・`invoke-swarm`(スウォームを起動)などの型があり、コンダクターはこれらに従って動きます。決定はエンジン、実行はコンダクター、進捗は状態ファイルと監査に刻まれ、あなたはステータスラインと承認ゲートを通してループ全体を見通せる — この透明性が、AI 駆動でありながら人間が主導権を握り続けられる理由です。

## AI-DLCを支える要素

AI-DLC を支えるのは、いくつかの意図的な設計です。まず **32 のステージと 11 のエージェント**。エージェントは細分化された多数の専門家ではなく、複数のステージ・フェーズにまたがって働く「広く有能な 11 のペルソナ」として定義されます。狭い専門家を大量に並べると、ウォーターフォールの引き継ぎの連鎖を機械の側で再現してしまうためです。同じアーキテクト・エージェントが立案フェーズでも構築フェーズでも設計を主導することで、文脈が自然に引き継がれ、明示的な引き継ぎ資料が不要になります。この「Small Mob, Broad Agents(小さなモブ、広いエージェント)」という発想は、3〜5 人のモブが一機能を丸ごとカバーする人間チームの姿を写したものです。

次に **ハーネス中立なコア**。方法論の本体はハーネス非依存の `core/` に一度だけ書かれ、各ハーネス向けの配布物(`dist/<harness>/`)はそこから生成されます。編集するのは常にコアであり、生成物(`dist/`)を手で書き換えることはありません。決定的なエンジン(状態機械、監査ログ、並列エージェントを調停するレフェリー)は、どのハーネスでもバイト単位で同一です。差が出るのは「見え方」の殻(シェル)だけであり、「one core, many harnesses(一つのコア、多数のハーネス)」という標語がこれを表しています。

さらに **適応的な深度とスコープ**。すべてのタスクに 32 ステージすべてが必要なわけではありません。9 つのスコープ(enterprise から workshop まで)と 3 段階の深度が、どのステージをどれだけの詳しさで実行するかを決めます。同じエンジンが、使い捨ての PoC と規制産業向けのエンタープライズ展開の両方を動かします — 違いは「より多くのステージを、より深く」走らせるかどうかだけなのです。

加えて **トレース可能な成果物と知識の二層構造**。成果物はすべて版管理された Markdown として蓄積され、意図から本番までの完全な意思決定記録を残します。68 種類の監査イベントが ISO タイムスタンプ付きで記録され、企業レベルのトレーサビリティを支えます。知識は二つの層に分かれます — フレームワークに同梱される方法論知識と、チームが自分たちで管理するドメイン知識です。そして、人間の修正を永続的な振る舞いへと変える「学習ループ」により、一度直した指摘が次のワークフローで自動的に効くようになります。

最後に、AI-DLC の振る舞いを形づくる **ルールとセンサーの制御ループ** に触れておきましょう。**ルール(Rule)** は、作業の前に適用される「立っている決定」で、org → team → project → phase → stage という厳格加算的な 5 層チェーンで解決されます。広い層が狭い層に上書きされることはなく、適用可能なルールはすべて文脈に現れます。**センサー(Sensor)** は、成果物への書き込みに対して発火する決定的な検査で、助言的な監査行を残します(ワークフローをブロックはしません)。ルールがフィードフォワード、センサーがフィードバックの半分を担い、両者でステージを操舵・検証します。そして **学習ループ** は、ステージ中の観測を `memory.md` に記録し、承認ゲートでそれらを提示して、あなたが残すと決めたものを恒久的な「プラクティス」としてスペースのメモリ層へ書き込みます。人間が修正を入れれば、それが組織レベルかプロジェクトレベルかに分類され、二度と同じ過ちを繰り返さないためのガードレールとして残る — この自己学習こそ、AI-DLC が「速さを保ったまま、文脈・根拠・制御を守る」という目標を実現している核心です。

こうした要素は、どのハーネスでも同じように働きます。AI-DLC は「one core, many harnesses」の名のとおり、Claude Code、Kiro IDE、Kiro CLI、Codex CLI といった複数の CLI に同一のコアを展開します。導入は、対象ハーネス向けの配布物(たとえば `dist/claude/.claude/` と `dist/claude/aidlc/`)をプロジェクトへコピーするだけです。どのハーネスも共通して、TypeScript のフックやツールを走らせる `bun` を前提とし、AWS Bedrock 上で動きます。起動子は Claude Code や Kiro では `/aidlc`、Codex では `$aidlc` です。導入後は `/aidlc --doctor` で環境の健全性を点検し、`/aidlc <説明>` で最初のワークフローを始められます。加えて、`--status` で現在地を確認したり、`--stage` や `--phase` で任意のステージ・フェーズへジャンプしたり、`--scope`・`--depth` をワークフローの途中で変更したりといった CLI ユーティリティも用意されています。方法論・エージェント・ゲートといった中身は共通で、違うのは「そのハーネス上でどう見えるか」だけ — この一貫性こそ、AI-DLC がツールを選ばずに同じ規律を提供できる理由です。

まとめると、AI-DLC とは「AI 支援開発を、5 フェーズ・32 ステージの再現可能でトレース可能な流れへと構造化する方法論」であり、その実装は、決定的なエンジンとオーケストレーションを担うコンダクターのループ、11 の広いエージェント、承認ゲート、版管理された成果物、そして自己学習の仕組みから成ります。人間が各決定を承認ゲートで統制し、AI がそれを忠実に実行する — この「User decides, AI executes」を軸に、案件の重さに応じて深度を伸縮させながら、使い捨ての試作から規制産業向けのエンタープライズまでを一つのエンジンで扱います。次の単元では「なぜこの方法論が生まれたのか」を従来手法との対比で掘り下げ、その後の単元で Intent・Unit・Bolt・Small Mob といった基本概念、5 フェーズの全体像、そして承認ゲートと人間の監督へと進んでいきます。AI-DLC の全体像をここでつかんでおけば、以降の学習がぐっと立体的になるはずです。
