AI-DLC を使いこなすには、うまくいく道だけでなく、はまりやすい罠も知っておく必要があります。この単元では、仕様なきコード生成と過信、コンテキストの肥大化と compaction、承認ゲートの停滞と 3-Strike 離脱、そして state 破損・hook 不発・lock 残留といった運用トラブルと復旧を扱います。ここで「アンチパターン」として名指しできるのは公式ドキュメントが原則や症状として明示しているものに限り、それを超える現場の経験則は、断定を避けた補足として本文に置きます。落とし穴の多くは「ディスク上に残るものと、メモリ上にしかないものの区別」と「人間がゲートで統制を握り続けること」で回避できる、という筋が全体を貫きます。裏を返せば、AI-DLC のアンチパターンとは、この 2 つの前提を忘れたときに現れるもの — 会話の中だけで作業を進めて compaction で失う、ゲートを形骸化させて過信のまま進む、あるいは逆に完璧主義で膠着する — です。以下の 4 つの落とし穴を、それぞれ公式が用意した受け皿(原則・compaction 保全・3-Strike/HUMAN_TURN・症状表と `--doctor`)とセットで見ていきます。

## 仕様なきコード生成と過信の罠

最初の、そして最も根深い罠が「仕様なきコード生成」と「AI への過信」です。AI-DLC の方法論原則はこれに正面から答えを置いています。**Questions before assumptions(仮定の前に質問)** — 迷ったら訊く。不完全な回答は貧弱な設計を生む、と原則は述べます。ad-hoc な AI コーディングはプロジェクトが本物になるまでは動くものの、そこからプロンプト間で文脈がドリフトし、判断の理由が記録されず、モデルが頼んでもいないことを静かに実行し始めます。AI-DLC はこれに構造で対抗します — 各ステージに明確なオーナーがいて、次に進む前にすべての判断が承認ゲートを通り、訂正から学んで繰り返さないようにします。同じエンジンが使い捨ての PoC も規制のある企業導入も回し、違うのは「どれだけのステージを、どれだけの深度で走らせるか」だけです。

過信を抑えるもう一つの原則が **No emergent behavior(創発的振る舞いなし)** です。エージェントは規定のプロトコルに従い、承認メニュー・完了メッセージ・状態遷移は標準化されています。勝手な即興は設計上排除され、振る舞いは予測可能です。加えて **User decides, AI executes(ユーザーが決め、AI が実行する)** — あらゆる実質的な判断は、ユーザーがレビュー・修正・却下できる承認ゲートを通ります。**Adaptive depth**(単純なプロジェクトは重いステージを飛ばし、複雑なものは全面カバー)、**Traceable artifacts**(全ステージがバージョン付き Markdown を残し、完全な決定記録になる)、**Multi-role expertise**(各ステージをドメインエキスパートのペルソナが導く)、**Contradiction detection**(スコープ・リスク・技術の矛盾を横断チェック)も、過信の防波堤です。人間が訂正した振る舞いは恒久的なガードレールになり、その誤りは二度と繰り返されません — ガードレールは組織レベル(全プロジェクト)かプロジェクトレベル(このリポのみ)に分類されます。この自己学習の仕組みがあるからこそ、「同じ過ちを AI が繰り返す」という典型的な過信の帰結を、一度の訂正で断ち切れます。5 フェーズ(Initialization / Ideation / Inception / Construction / Operation)それぞれが明確な目的と成果を持ち、各ステージをドメインエキスパートのペルソナが導くのも、「賢いが文脈を持たない汎用アシスタント」に丸投げする ad-hoc なやり方とは対極の設計です。

ここから先、「どのステージで特に人間が踏ん張るべきか」「どこまで自律に任せるか」といった勘所はプロジェクトやチームによって異なり得ますが、原則としては「仕様を先に固め、ゲートで必ず人が目を通し、決定を残す」ことが、仕様なき生成という罠への一次防御になる、と言えます。生成された出力は行動前に必ずレビューする、という姿勢は、公式ドキュメントも繰り返し強調するところです。

過信の裏返しとして、逆の落とし穴 — 全ステージを重々しく回そうとする「過剰プロセス」— もあります。AI-DLC の **Adaptive depth** と 9 スコープはこれに答えます。すべてのタスクが 32 ステージ全部を要するわけではなく、スコープ(enterprise・feature・mvp・poc・bugfix・refactor・infra・security-patch・workshop)がどのステージをどの深度で走らせるかを決めます。bugfix に企画フェーズ一式を課すのも、enterprise 級の導入をゲート無しで流すのも、どちらも罠です。仕様を固める重さは作業の性質に合わせる — 「仕様なき生成」を避けることと「過剰な儀式」を避けることは、スコープと深度という同じダイヤルの両端だと捉えると、ちょうどよい厚みを選べます。

## コンテキスト肥大化と compaction

長いセッションで避けられないのが、コンテキストウィンドウの肥大化です。Claude Code はウィンドウが埋まると以前の会話文脈を自動で要約します — これが **compaction(コンパクション)** です。ここで何が保持され何が失われるかを正しく理解していないと、「AI が急に忘れた」ように見えて慌てることになります。

保持されるのは **ディスク上のもの** — レコードディレクトリの全アーティファクト、`aidlc-state.md`(ステージ進捗・スコープ・プロジェクト情報)、`audit/` シャード(判断と行動の全履歴)、`.aidlc-recovery.md`(ステージチェックポイント)です。失われるのは **メモリ上のもの** — 以前の会話文脈、ファイルに未書き込みの途中作業、そして Task ID(state から再構築)やエージェントのペルソナ文脈(エージェントファイルから再読込)です。compaction は長いセッションの正常な一部であり、ディスク上の state とアーティファクトが完了済みの作業の消失を防ぎます。つまり落とし穴は compaction そのものではなく、「重要な決定や作業をメモリに置いたままファイルに落とさないこと」にあります。

復旧は難しくありません。compaction の前に `validate-state.ts` フックが隠しファイル `.aidlc-recovery.md`(最終検証のタイムスタンプ・現ステージ名・state 有効性)を書き、次回 `/aidlc` 起動時にそれを `aidlc-state.md` と突き合わせます。「Current stage」が食い違えば、compaction による state 破損の可能性を警告します。手順としては、`/aidlc` を実行して state から位置を読み直し、リカバリのブレッドクラムが不一致を警告したら **Redo current stage** を選んで進行中だったステージを安全に再実行し、警告が無ければ **Resume from last checkpoint** で通常どおり続けます。resume 時には 4 つの選択肢(Resume / Redo / Jump / Start fresh)が提示され、それぞれ「何が保たれ何が失われるか」が明確です。

4 つの選択肢を具体的に見ると落とし穴の避け方が分かります。**Resume from last checkpoint** は進行中/次の pending ステージから続け、全アーティファクト・state・監査を保ち、失うのは前セッションのメモリ上会話だけ。**Redo current stage** は現ステージのアーティファクトディレクトリを消してチェックボックスを `[ ]` に戻し最初からやり直す — 他は保つが現ステージの途中作業は失う。**Jump to stage** は特定ステージへ飛び、間のステージを `[S]` にする(下流アーティファクト無効化の警告付き)。**Start fresh** は現 intent のレコードをアーカイブ(確認必須)して新しい intent を born する。compaction で「現ステージが途中だった」ときは Redo が安全、というのが定石です。落とし穴は、compaction 警告を無視して Resume を押し、半端な状態のまま進めてしまうこと — ブレッドクラムの不一致警告は、Redo を選ぶべきサインだと覚えておきます。落とし穴を避ける鍵は、やはり「途中作業は早めにファイルへ落とす」— メモリ上にしかない作業だけが失われる、という一点にあります。Codex ハーネスは PostCompact でワークフローのミッションを再注入するなど、ハーネスによって compaction 後の挙動に差がある点も、複数ハーネスを使うなら意識しておくとよいでしょう。compaction を「事故」ではなく「長いセッションの正常な一部」と捉え、要所要所でアーティファクトへ書き出す習慣をつけておけば、要約が走っても作業は失われません。逆に、重要な決定を会話の中だけに置いたまま長時間走らせるのが最大のアンチパターンで、これはディスクに落とすという一手で確実に防げます。state 破損の警告が出たときに Redo を選べる冷静さも、この「保持されるもの/失われるもの」の地図を頭に入れているかどうかで決まります。

## 承認ゲートの停滞と 3-Strike リビジョン離脱

承認ゲートは統制の要ですが、同じステージで完璧を求めて回り続けると停滞します。AI-DLC は無限リビジョンループを防ぐ **3-Strike リビジョン脱出ハッチ** を持ちます。標準ゲートは 2 択(Approve / Request Changes)ですが、同じステージで 3 回以上 Request Changes を出すと 3 番目の選択肢 **Accept as-is** が現れ、現行版のアーティファクトをアーカイブしてワークフローを進めます。

その現れ方は段階的です。1 回目は標準 2 択、2 回目は 2 択に「あと 1 回リビジョンすると Accept as-is が使える」という注記が付き、3 回目以降で Accept as-is 付きの 3 択になります。リビジョンカウントは次のステージに移るとリセットされます。「完璧が善の敵」になるのを防ぐ設計で、停滞したら現行版を受け入れて前に進む逃げ道が用意されている、と理解しておくとゲートで固まりません。無限に直し続けるより、受け入れて次へ進み、必要なら後で戻る方が健全な場合が多いのです。

もう一つの「停滞」は、単にゲートで待っているだけの状態です。トラブルシューティングの指針は明快で、プロンプトが出たら応答をタイプすること。進めない・飛ばしたいときは `/aidlc --stage <target>` で別ステージへジャンプでき、間のステージは `[S]`(スキップ)と記録されます(下流の成果物が陳腐化しうる警告付き)。なお承認ゲートは **本物の人間が行動したこと** を要求します — 人間のプロンプトが監査台帳に `HUMAN_TURN` を記録し、前回のゲート解決以降にそれが無ければ approve は拒否されるので、自動運転のモデルが人間不在で承認を捏造することはできません。ゲート widget が人間ターンを記録しないハーネスでは、一度「approve」と短く打てば記録が残ります。Ideation / Inception のゲートでは、スコープルーティングで飛ばした関連ステージを呼び戻す条件付きの追加選択肢も現れます。ここでの落とし穴は「延々と Request Changes を繰り返して前に進めない」ことと、逆に「人間が本当に見ずに通す」ことの両方で、3-Strike と HUMAN_TURN ガードはその両端を抑えます。

訂正がそもそもどう起きるかも押さえておくと、ゲート停滞を避けやすくなります。ステージが入力を集めるとき、エージェントは 3 つの対話モード — Guide Me(構造化された質問に答える)・Edit File(質問ファイルを直接編集)・Chat(自由な議論から決定を抽出)— を提示し、どのモードでもステージの質問ファイルが決定の正本になります。モードは途中で切り替えても進捗は失われません。要件がまだ固まっていないなら Chat で探索し、明確なら Edit File で書き下す、という使い分けが、後段での手戻り(=リビジョンループの原因)を減らします。承認のたびに `Progress: 13/32 overall | 3/7 IDEATION stages complete. Next: ...` のような進捗行が出るので、自分が全体のどこにいて次に何が来るかを把握でき、闇雲な却下を避けられます。ゲートは統制のための「立ち止まる場所」であって「膠着する場所」ではない、という感覚が大切です。

## state 破損・hook 不発・lock 残留の運用トラブルと復旧

最後は運用トラブルとその復旧です。トラブルシューティングは症状で引けるようになっています。まず **hook が発火しない** 症状 — ファイル書き込み後も `audit/` シャードにエントリが出ない・サブエージェント完了ログが出ない。最頻の原因は `bun` が未インストールか PATH に無いことです。11 個の TypeScript フックはすべて `bun` を要するので、`bun --version` で確認し、非対話シェルの PATH(`~/.zshenv` や `~/.bashrc` — `~/.zshrc` だけでは不足)に通します。もう一つの原因は hook 未設定で、`.claude/settings.json` の `hooks` ブロックに `PostToolUse`・`PreCompact`・`SubagentStop`・`Stop`(＋ `SessionStart`/`SessionEnd`)があるか確認します。アップグレードでこれらの置き場所が変わった版を取ったなら、出荷される `settings.json` の hooks ブロックを再コピーします。

次に **state 破損**。`validate-state.ts` フックは compaction のたびに `## Stage Progress` と `## Current Status` の 2 セクションを確認します。手動修復では、intent の `aidlc-state.md` を開き、Project Information・Scope Configuration・Workspace State・Stage Progress・Current Status・Session Resume Point の各セクションがあるか確かめ、`.claude/knowledge/aidlc-shared/state-template.md` テンプレートと比べて欠けたセクションを `audit/` 履歴の値で埋め直します。**サブエージェントのタイムアウト**(Workspace Detection・Reverse Engineering・Code Generation)は組み込みのリトライ(縮小コンテキストで自動再試行 → 失敗ならインライン実行かスキップ&後で再訪)で扱われ、`/aidlc` を再実行すれば `[-]`(進行中)を検知して resume/redo を提案します。

そして **lock 残留**。監査フックは `mkdir` ベースのロックで並行書き込みを防ぎますが、フックが中断されるとロックディレクトリが残ることがあります(症状: フックが一瞬ハングしてスキップ、以降の監査エントリが書かれない)。ロックはシステムの一時ディレクトリ(`/tmp/` など)に `.aidlc-audit-<hash>.lock` / `.aidlc-subagent-<hash>.lock` として作られます。ワークフローが動いていないときなら `rm -rf /tmp/.aidlc-audit-*.lock /tmp/.aidlc-subagent-*.lock` で安全に消せます(ロックは一時的で、各フック呼び出しで再作成されます)。監査ログが肥大したら `audit/<host>-<clone>.md` をアーカイブへ rename すれば、次回起動で新しいシャードが作られます — 監査シャードはルーティングに読まれないので安全に退避できます。困ったときに最初にやるべきは `/aidlc --doctor` で、これは読み取り専用で、前提(`bun`)・hook 可用性(11 個すべてが存在するか)・プロジェクト構造・ワークスペースシェルの準備・state/audit 整合・hook ハートビート・グラフ整合(循環なし・全エントリにファイル)・9 スコープの検証・ステージスキーマとグラフ参照・スコープ間のキーワード重複を一括点検します。fresh なシェルでは何も作らないので、何かおかしいと感じたら最初に叩くコマンドとして安全に使えます。運用トラブルの多くは、この症状ベースの引き方と `--doctor` で切り分けられる、というのがこの単元の締めです。

トラブルシューティングには症状→即応の早見表もあります — 監査エントリが出ない → `bun` が PATH にあるか確認、state 破損 → `--doctor` でテンプレートと比較、ゲートで停滞 → 応答をタイプするか `--stage` でジャンプ、compaction 発生 → `/aidlc` で checkpoint から resume、監査ログ肥大 → `audit-YYYY-MM.md` へ rename、hook がハング → 一時ディレクトリの stale lock を除去、サブエージェントタイムアウト → `/aidlc` で再試行かインライン実行。症状から入れば、原因の当たりを素早くつけられます。`--doctor` はさらに、常にパスする 2 つの advisory 行を出します — **Rule drift**(team/project ルールが populated な org 見出しと重なり矛盾レビューを促す)と **Paired sensor coverage**(`pairing:` を持つルールの named sensor がステージに解決するか)。どちらも読み取り専用で exit code を変えません。黒帯としての心構えは、これらの落とし穴を「起きてから慌てる」のではなく、「ディスクに落とす・ゲートで人が見る・症状表と `--doctor` で切り分ける」という定石で先回りすることです。仕様なき生成・過信・コンテキスト消失・ゲート膠着・運用トラブル — どれも AI-DLC が原則と仕組みで受け止める設計になっており、その受け皿を知っていること自体が最大の予防策になります。最後に `--doctor` の性格を補っておくと、これはフルパスで exit 0、いずれかの失敗で exit 1 を返し、レポートはどちらでも stdout に書きます。intent が存在すれば `HEALTH_CHECKED`(および `GUARDRAIL_LOADED`)の監査行を残しますが、intent がまだ無い fresh なシェルでは何も作りません。だから「何かおかしい」と感じた瞬間に、まだワークフローを始める前でも安全に叩ける、最初の一手として設計されています。wired されているのに存在しない hook は loud に失敗するので、hook 不発の切り分けにも有効です。症状で早見表を引き、`--doctor` で全体を点検し、必要なら state テンプレートと突き合わせて手で直す — この 3 段構えを身につけておけば、運用フェーズの大半のトラブルは落ち着いて復旧できます。
