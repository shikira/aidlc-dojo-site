Inception でユニットと依存 DAG、そして配送計画ができたら、いよいよ **Construction フェーズ**で「もの」を作ります。Construction は設計成果物を、動いてテストされたソフトウェアへ変える段で、**Bolt(ボルト)** という単位でレビュー可能なスライスを刻みながら進みます。Construction は 5 フェーズの 4 番目で、2.8 Delivery Planning が生んだ実行計画に駆動され、どのステージが走り/飛ばされ、どの順でユニットが作られるかはその計画が決めます。この単元では、Bolt-by-Bolt の全体像、ウォーキングスケルトンと ladder prompt、並列バッチと halt-and-ask、そして convergence と swarm の位置づけを実践の視点で押さえます。

## Bolt-by-Bolt で進める Construction

なぜ Bolt 単位なのか、には経緯があります。かつて Construction はユニットごとにステージを回し、各ステージ後に承認ゲートを置いていました — 3 ユニットなら、テスト済みコードが 1 行出る前に 15 個のゲート。顧客はこれを「子守り(babysitting)」と呼びました。次の修正は、全ユニットの質問・設計・コード生成をまとめて最後に一度だけレビューする方式でしたが、今度は振り子が逆に振れ、15 ユニットの実行が 15,000 行を一度のゲートに積み上げる事態になりました。一度のレビューで検証するには多すぎたのです。

現在の姿はその中間です。Construction は **Bolt by Bolt** で走ります。1 つの Bolt は、あるユニット(または依存で結ばれた小さなユニット群)に対するステージ **3.1〜3.5** の 1 パスです。最初の Bolt が**ウォーキングスケルトン** — アーキテクチャを証明する、最小の端から端まで動くスライス。それが出荷されると **ladder prompt** が一度だけ発火し、その回答が状態に記録されて残りの Bolt すべてを統べます。ステージ **3.6 Build and Test と 3.7 CI Pipeline は、全 Bolt の完了後に一度だけ**、コードベース全体に対して走ります(Bolt ごとではありません)。この形は、早期の自信チェックポイントと意図的な自律の選択を、2.8 が計画したサイズのレビュー可能なスライスとともに与えます。要するに、ステージごとに 15 個のゲートで子守りさせるのでもなく、15,000 行を一度に積み上げるのでもなく、「Bolt という程よい粒度のスライスを刻む」ことで、レビューの負荷と自律の速さのバランスを取っているのです。

Construction は 2.8 の `bolt-plan.md`(Bolt の並び+ウォーキングスケルトンの印)と 2.7 の依存 DAG に駆動されます。各 Bolt の中では、3.1〜3.4 の質問を Bolt のユニット横断で **QUESTION-ONLY モード**にまとめて集め、Bolt レベルの回答ゲートで確認してから、**ARTIFACT-ONLY モード**で設計成果物を生成し、3.5 Code Generation をユニットごとに Task 委譲で走らせます。`code-generation.md` に本来ある per-Unit の承認ゲートは、通常の Bolt 実行中はエンジンによって**抑制**され、単一の Bolt レベル(またはバッチレベル)ゲートが置き換えます(per-Unit ゲートは `/aidlc --stage code-generation` の直接呼び出し時には残ります)。7 ステージのうち per-Unit 対象は 3.1〜3.5 で、リードは 3.1 Functional Design が aidlc-architect-agent、3.2 NFR Requirements・3.3 NFR Design が aidlc-architect-agent、3.4 Infrastructure Design が aidlc-aws-platform-agent、3.5 Code Generation が aidlc-developer-agent、末尾一括の 3.6 が aidlc-quality-agent、3.7 が aidlc-pipeline-deploy-agent です。

各設計ステージ(3.1〜3.4)は QUESTION-ONLY と ARTIFACT-ONLY の 2 モードを持ち、Bolt 内では「まず Bolt のユニット横断で質問だけを一気に集めて Bolt レベルの回答ゲートで確認し、その後に成果物だけを生成する」という流れになります。質問収集の一括化は、回答が結局ユーザーを通して直列化されることを踏まえた設計です。3.6 Build and Test はコードベース全体に対して一度だけ走り、既存テストのベースラインと突き合わせて回帰を検出します。3.7 CI Pipeline は条件付きで、CI 設定の新規作成や大幅な変更が要るときに一度だけ走ります。つまり Construction は「Bolt ごとに設計〜コード生成を回し、最後に全体のビルド・テスト・CI を一度だけ束ねる」という二段の構造を持ちます。

## Walking Skeleton と ladder prompt

**ウォーキングスケルトン**は Construction の最初の Bolt — すべての統合点を貫く、最も薄い端から端までのスライスです。これは**常にゲート付き・対話的**で、全体の形をあなたが確認してから残りの Construction が走ります。組織の既定(`org.md` の `## Walking Skeleton`)では、greenfield スコープ(`mvp`・`enterprise`・`feature`・`poc`・`workshop`・`infra`)でスケルトン Bolt が最初に走り、Bolt 1 は単独・ゲート付きで、あなたが承認してから残りが進みます。一方、増分スコープ(`bugfix`・`refactor`・`security-patch`)では**スケルトン儀式は飛ばされます** — 既存コードベースにブートストラップするものが無いため、最初の Bolt も他と同じように走ります。

スケルトンが承認された**直後に一度だけ**、**ladder prompt** が発火します。

```
The walking skeleton shipped. How should the remaining Bolts run?
  ▸ Continue autonomously
  ▸ Gate every Bolt
```

あなたの回答は `aidlc-state.md` に **`Construction Autonomy Mode`**(`autonomous` または `gated`、プロンプト前は `unset`)として記録され、このワークフローの残り Bolt すべてを統べ、セッション再開でも尊重されます。ladder prompt はワークフローにつき厳密に一度だけです。この形は、早期の自信チェックポイント(スケルトンで全体の形を確認)と、意図的な自律の選択(残りをどう回すか)を、一度ずつ与えます。スケルトンが常にゲート付きなのは、アーキテクチャの妥当性を人間が確かめる前に大量のコードが積み上がるのを防ぐためで、白帯で学んだ「User decides, AI executes」がここでも効いています。

ここで重要なのは、autonomy が**姿勢(標準の推奨)**であって**許可そのものではない**という切り分けです。フレームワークは決定・知識・判断を「それが何の種類か」で分けます — 決定論はツールに、知識はエージェントに、判断は人間に、という三分割です。チームは `team.md` や `project.md`(いずれもスペースのメモリ層 `aidlc/spaces/<space>/memory/` にある)の Walking Skeleton 見出しの下にルールを書いて推奨を形作れます。たとえば自律 Construction に不慣れなチームは、保守的な姿勢として「私たちのチームがクリーンな自律バッチを 3 回出荷するまでは、ladder prompt での推奨は『gate every Bolt(毎 Bolt をゲート)』とし、レビュアーは次の Bolt が始まる前に各 Bolt の diff を見る。収束チェックの信頼性が証明されたら、この既定を見直す」と書きます。逆に信頼済みのスコープへハンズオフへ卒業するチームは、鏡写しに「この codebase の `feature` スコープでは、ウォーキングスケルトンが緑になったら ladder の推奨は continue autonomously とする」と書きます — 同じファイル・同じ見出し・反対の推奨です。

こうしたチームの散文は org の既定に**積み重なり**(スケルトン先行/儀式スキップの切り分けは不変)、あなたのチームの散文が ladder prompt の場でエージェントの文脈に加わって推奨を形作ります。ただし実際にこのプロジェクトを自律で回すかどうかの判断は、ゲートにいる人間に残ります。あなたが設定するのは**既定とガイダンス**で、許可(grant)は人間に残る — 決定論・知識・判断の線が 1 つのプロンプトを貫きます。ルール編集は次のワークフローのコンパイル境界で効き、進行中の実行を遡って変えることはありません。org 自体はフレームワーク出荷・継承なので、あなたは触りません。

## 並列バッチと halt-and-ask

Construction のスループットを上げるのが**並列バッチ**です。2 つ以上の Bolt が依存充足を共有し(たとえば B と C がともに A だけに依存)、互いに依存しないとき、それらは 1 つの**バッチ**として同時に走ります。バッチ末尾の単一ゲートがバッチ内の全 Bolt を覆います。

どのユニットが並列に走れるかは、上流の 2.7 Units Generation が捉えた依存構造で決まります。2.7 が生む `unit-of-work-dependency.md` の中の必須 YAML エッジブロックが、各ユニットとその `depends_on` を列挙し、コンパイラがそれを `runtime-graph.json` の `bolt_dag` ノードへ読み込みます。このノードはエッジブロックが well-formed かつ非循環のときだけ現れ、`batches`(依存が先行レベルで満たされるトポロジカルレベル=互いにエッジの無いユニット群)を持ちます。粗いユニットで相互依存が少なければ多くのユニットが同じバッチに入って並走し、深く連鎖した依存は作業を多数の小さなバッチに直列化します。したがってハーネスエンジニアは「何が並列化するか」を間接的に、しかし確実に形作れます — チームガイダンスやアーキテクトが読むルールを通じてユニット分解を粗く保てば、より多くが同時に走ります。分解そのものは人間とエージェントの知識的判断であり、そのトポロジーをコンパイラがバッチへ変える、という役割分担です。

コンダクター(稼働中の `/aidlc` セッション)は、並列 Bolt を「1 つのアシスタントターンで複数の `Task` 呼び出しを発行する」ことで捌きます — Claude Code 組み込みの並列性が、各 Bolt の Code Generation ステージを同時に走らせます。質問収集と設計成果物の生成は Bolt ごとに直列のまま(安価であり、回答は結局ユーザーを通して直列化されるため)です。監査イベント(`BOLT_STARTED`・`BOLT_COMPLETED`)は `Batch=N` フィールドを持ち、兄弟 Bolt がログから復元できます。なお並列バッチには複数の `Task` 対応サブエージェントスロットが要り、gated モードならバッチ末尾の単一ゲートが全 Bolt をまとめて覆い、autonomous モードならそのゲートも省かれます。バッチという単位のおかげで、依存的に安全な範囲だけを同時に進めつつ、レビューは 1 回で済む、という並列性と監督のバランスが取れます。

そして Construction のもう一つの規律が **halt-and-ask on failure(失敗時の停止と問い合わせ)** です。**失敗は、autonomy モードに関わらず常に Construction を止めます**。これが自律モードで唯一割り込む場面です。単独 Bolt が失敗すれば即座に停止し、**retry**(その Bolt だけ再実行)・**skip**(`[S]` と印して継続 — 依存する Bolt も失敗しがち)・**abort**(Construction 全体を停止)を尋ねます。並列バッチの中で 1 つの Bolt が失敗し他が成功したときは、コンダクターはバッチ全体の完了を待ち、成功した Bolt の成果物(`[x]` ステータス)をディスクに保ったまま、失敗した Bolt についてのみ同じ retry / skip / abort を提示します。ハンズオフモードは幸せな道のゲートを省くだけで、失敗時の停止は常に鳴り響きます。この一点は、白帯で学んだ「自律モードを選んでも失敗は必ず人間に判断を戻す」という原則の、Construction における具体的な現れです。速さのために happy-path のゲートを省いても、判断が要る場面(失敗)では必ず人間へバトンが返る — この非対称性が、ハンズオフと無責任の違いを作っています。

## Convergence と swarm

Construction をユニット横断で大きくファンアウトさせるのが **swarm(スワーム)** です。ここで最も大切な原則は、**ワーカーの「収束した」という主張をフレームワークは決して鵜呑みにしない**ことです。権威ある信号は、あなたのプロジェクト**自身のチェックコマンド**です — レフェリーがユニットごとに走らせ、終了コード `0` が本物の収束、それ以外は「まだ」を意味します。ハーネスエンジニアが自律 Construction のために保証すべき最重要事項は、プロジェクトが本物のチェックコマンド(`npm test`・`pytest`・build+lint スクリプトなど)と**保護された仕様ファイル**を実際に持っていることです。レフェリーは指定された `--test-file` をフォークした git ベースラインと突き合わせて改ざん検出できるので、ワーカーが「done」の定義であるテストをこっそり弱めて赤を緑に見せることはできません。常に通るチェックや空の仕様は、swarm にゴム印を渡すのと同じです。

swarm が物理的にどうファンアウトするかは、環境変数 **`AIDLC_USE_SWARM`** で選ばれる**オペレーターのつまみ**です(`.claude/` のデータファイルでも `settings.json` でもなく、ファンアウト時にコンダクター側で読まれます)。未設定または `"1"` 以外なら **subagent floor** — コンダクターが 1 メッセージで N 個の並列 `Task` を発行(ユニットごとに 1 つ)。`"1"` なら **inline Dynamic Workflow** — コンダクターが per-Unit パイプラインと反復上限を持つ `Workflow` を書きます。`"1"` でも Workflow ツールが使えなければ、floor へ**loud-degrade(声を上げて縮退)**し、`--degraded-from ultracode` を渡してレフェリーが `SWARM_DEGRADED` を発します。どちらのドライバーも同じ 5 つの per-Unit ステージを走らせ、同じプロジェクトチェックに対して収束します — 違いは並列作業の配り方だけです。暴走の歯止めはハーネスの Stop-hook 天井(この自律 Construction 経路では既定 8 ブロック、対話既定は 2、`CLAUDE_CODE_STOP_HOOK_BLOCK_CAP` で上書き)にあり、swarm ツールの外側に置かれます。なお Kiro・Codex では Workflow ツールが無いため、`AIDLC_USE_SWARM=1` は常に floor へ loud-degrade します — つまり swarm の「速い配り方」はハーネス依存でも、収束の判定と失敗時の停止という**意味論はどのハーネスでも同一**です。

そして 1 つの判断だけはドライバーと共に動きません — **失敗は autonomy モードに関わらず常に停止して人間に戻す**。レフェリーの `finalize` が exit-2 の封筒を返すと、コンダクターはバトンを人間へ返します。レフェリー `aidlc-swarm.ts` は状態を持たない `prepare` / `check` / `finalize` で、claim された全ユニットをマージ前に再検証し(嘘つきコンダクター対策)、マージバックを直列化し、6 つの `SWARM_*` 監査イベントを発します。並列化できるのは、Inception の `units-generation` が走るスコープ(`enterprise`・`feature`・`mvp`・`workshop`)だけである点も押さえておきましょう — 増分スコープや poc / infra は依存エッジブロックも `bolt_dag` も持たず、Construction は単一パスで走り、swarm がファンアウトする対象がありません。ハンズオフ Construction は「全スコープの性質」ではなく「多ユニット greenfield スコープの性質」なのです。

Construction のつまみを整理すると、「誰の持ち物か」で三つに分かれます。**あなた(ハーネスエンジニア)** が握るのは、自律の姿勢(`team.md`/`project.md` のルール=データ)、何が並列化できるか(`units-generation` と依存 DAG=データ)、収束の判定に使うプロジェクト自身のチェックと保護された仕様(データ+プロジェクト設定)。**人間** が握るのは、このプロジェクトの実際の自律許可(ladder prompt)。**オペレーター** が握るのは、swarm ドライバーの選択(`AIDLC_USE_SWARM` 環境変数)。そして**ツール**が握るのが、収束の判定・マージバック・監査(`aidlc-swarm.ts`)。データを編集して形作れるのは前三者のうちあなたの三行だけで、コードには触れません。この分割を意識すると、「存在しない設定を書こうとしてしまう」失敗を避けられます — たとえば自律の許可そのものは設定ファイルではなくゲートの人間の判断であり、ドライバー選択は環境変数であって `.claude/` のデータではありません。ここまでで、Inception の計画が Bolt となって回り、ウォーキングスケルトンで形を確かめ、autonomy モードで残りを進め、失敗すれば必ず人間に戻り、swarm がプロジェクト自身のチェックに対して収束する、という Construction の実践像が一巡しました。次は、こうした振る舞いを標準として形作るステアリングルールの構造へ進みます。
