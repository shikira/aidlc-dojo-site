エージェントの振る舞いを形作るのが**ルール(ステアリング)** です。ルールは「どう働くか」を決める永続的な散文の指示で、AI-DLC の要になります。この単元は現行の **v2** を主軸に、5 層のルールチェーンと学習ループ・Sensors・Rule System の実装を押さえたうえで、レガシーである **v1(v1.0.1)** の 2 層構造と対比します。current=v2 / v1=legacy の枠で、両系の違いを理解するのが狙いです。

## v2 の5層ルールチェーン

v2 では、ルールはスペースのメモリ層 `aidlc/spaces/<space>/memory/` に Markdown ファイルとして置かれます。ワークスペースルートにある単一の手編集可能なセットで、各ハーネスがネイティブなインクルード(Claude の `@`-import スタブ、Kiro の resources glob、Codex の `AIDLC_RULES_DIR`)で読み込みます。

```
aidlc/spaces/<space>/memory/
├── org.md                 # フレームワーク + 組織横断の既定
├── team.md                # チームが是認した実践
├── project.md             # このプロジェクトの特化
└── phases/
    ├── ideation.md
    ├── inception.md
    ├── construction.md
    └── operation.md
```

ファイルはスコープごとに名付けられます — `org.md`(トランクベース開発・テスト方針・ウォーキングスケルトン方針などのフレームワーク既定)、`team.md`(チームが是認した内容)、`project.md`(このプロジェクト固有)、そして `phases/<phase>.md`(各フェーズの全ステージに適用 — 例: inception フェーズルールは全アーキテクチャ判断に最低 2 案の代替を要求)。各ファイルは `## Way of Working`・`## Testing Posture`・`## Deployment`・`## Code Style` のような話題見出し下の素の散文で、あなたが読み、手で編集でき、フレームワークが学習ループを通じて書き込みます。読み手(エージェントやオーケストレーター)は見出しを完全一致ではなく**話題(topic)で照合**し、team → org → ハードコード既定という順にフォールバックして、最初に見つかった非空セクションを使います。**ファイル内に `scope:` フィールドは無く、スコープはファイル名から導かれます**。

ルールはワークフロー開始時に **5 層のチェーン**で解決されます。

```
org → team → project → phase → stage
```

このモデルは **strict-additive(厳密加算)** です。適用可能なルールはすべてエージェントの文脈に現れ、実行時に黙って落ちたり上書きされたりしません。org の既定・team の実践・project の特化は**連結**され、該当する phase ルールはステージが既に宣言している `phase:` によって付きます(第 5 層の per-stage ルールは将来用に予約)。これは以前の版からの意図的な変更で、`overrides:` ブロックも `enforcement:` キーワードももうありません。すべての層が同時に存在し、**衝突はルールが書かれる時点(実行時より前)に捕まえられます**。たとえば inception フェーズルールは「すべてのアーキテクチャ判断に最低 2 案の代替を記す」ことを要求し、これは inception を宣言したどのステージにも自動で効きます。チェーンは**ワークフロー開始時に一度だけ**解決されます — フレームワークがステージ定義・ルール・センサーを 1 つのグラフへコンパイルする、まさにその瞬間です。ワークフローの間、エージェントは解決済みのビューを読み、実行中にチェーンを歩き直すことはありません。

この規律はネットワークのルーターから借りた **planes(平面)** モデルで説明されます — **control plane**(何が走るべきかのスキーマ=ステージ定義・ルール・センサー。一度だけ走るので遅くて賢くてよい)、**data plane**(実際の実行=ステージ実行・エージェント呼び出し・レコードディレクトリのファイル。解決済みの答えを読むだけ)、**management plane**(`/aidlc --doctor`・監査ログ・`CLAUDE.md` による観測と設定)。control plane がワークフロー開始時に一度だけルールとセンサーをグラフへコンパイルし、data plane が残りの実行でそれを読む — これが「学習は次のワークフローから効く」挙動の根拠です。

## 学習ループと Sensors

**学習ループ**は、一度きりの訂正を永続的なルールへ変える仕組みです。あなたがワークフロー中にエージェントを訂正し、そのステージの承認ゲートで訂正を確認すると、フレームワークがそれを保存し、同じ訂正が二度と要らなくなります。ループには 4 つの見える瞬間があります — ステージ中の日記、ゲートでの候補提示、あなたの確認、そして次回のための書き込み。

ステージが走る間、フレームワークは観測ログ `<record>/<phase>/<stage>/memory.md` を保ちます。自動作成・自動維持され、手編集はしません。項目は 4 つの標準見出し — **Interpretations**(散文が曖昧だった箇所の解釈)・**Deviations**(意図的に逸れた箇所とその理由)・**Tradeoffs**(検討した代替と選択理由)・**Open questions**(次回までに確認したいこと)の下に、タイムスタンプ付きで落ちます。言語モデルがこのループで担う唯一の仕事は、この日記を書くことだけです — 数える・提示する・振り分ける・書き込む、はすべて決定的ツールかあなたの明示的選択が行います。

各承認ゲートの前に**ゲート儀式(プロトコルの §13)** が走り、2 つの源から候補を集めて 1 つの確認リストにします — 日記の各行を**逐語で**提示するものと、「次回のために足すことは?」と必ず尋ねる自由記述チャネル(あなたは 4 見出しのどれに属すかだけ選ぶ)。あなたはキープしたい候補にチェックを付けます。**行き先はあなたが選ばず、見出しが決めます** — Interpretations・Deviations・Tradeoffs は実践として `project.md` に落ち(確認された学習はすなわち実践)、Open questions は昇格しません(研究項目でありルールではないため)。既定スコープは **project** で、ワンクリックで `team.md` へ広げられますが、org へ広げる道はありません(org ルールは別プロセスで著される)。

ディスクへ落ちる前に `org.md` に対する**セクション単位の admission 衝突チェック**が走り、矛盾があればゲートが止まって org の一文を引用し、あなたは修正・スキップ・エスカレーションを選びます(上書きの道はありません)。ANZ 銀行プロジェクトの例が分かりやすい — Sam が「transaction」を書き、エージェントが DB トランザクションと解釈、Sam が「銀行取引(決済)」と訂正。日記が `## Interpretations` に記録し、ゲートで逐語提示され、Sam がキープ。org 衝突チェックを通って `project.md` へ書かれ、`RULE_LEARNED` が監査される。**現在のワークフローは変わらず**(この実行では既に会話でエージェントを訂正済み)、次回 `/aidlc` のコンパイルで初めて「transaction=決済」が全ステージの文脈に入ります。これはルーターが BGP から得る安定性と同じ性質です — 経路はパケット飛行中に再計算されず、AI-DLC はワークフローの途中で再コンパイルしません。org ポリシーが後から変わった場合の post-write ドリフトは `/aidlc --doctor` が助言的に(ファイル・セクション・矛盾する org 文を名指しして)surface します。

**Sensors(センサー)** はルールと対をなす決定的な検証です。ルールが「ユーザーストーリーは Given/When/Then 形式で」と散文で言うのに対し、センサーは必要な見出しが実際にファイルにあるかをバイト単位で検証します。エージェントがステージ出力を Write/Edit すると PostToolUse フックが該当センサーを走らせます — マッチングはファイル形状(たとえばコード品質センサーは `**/*.{ts,js}` のみ)によります。結果はこの版では **advisory(助言)** で、失敗しても監査行と詳細ファイル(欠けた見出し・未参照の上流成果物・lint エラーを名指し)を残すだけで、ゲートを止めません。フレームワークは 4 つのセンサー(`required-sections`・`upstream-coverage`・`linter`・`type-check`)を同梱します。ルールが feedforward(前送り)、センサーが feedback(後送り)の半分で、二つ合わせて control loop(制御ループ)を成します。

キープした学習が「ルール」ではなく「センサーのバインディング」のとき(あるステージ出力に新しい決定的チェックを走らせたいとき)、フレームワークは二重書き込みを原子的に行います — センサーのマニフェストを scaffold し、そのセンサーの id を originating ステージのインポートリストへ追記します。日記・ゲート確認・結果のファイル書き込みは、それぞれ監査行(`RULE_LEARNED` または `SENSOR_PROPOSED`)を残すので、ルールが黙って入ることは決してありません。センサー活動は `audit/` シャードに `Sensor Fired`・`Sensor Passed`・`Sensor Failed` として現れ、失敗行は具体的なギャップ(欠けた見出し・未参照の上流・lint エラー)を列挙した詳細ファイルへリンクします。自分でセンサーを足すには `.claude/sensors/` にマニフェストを著し、走らせたいステージへ id を加えます。

## Rule System の実装詳細

スキーマ側から見ると、ルールの**スコープはファイル名から導かれ**ます — `org.md`→org、`team.md`→team、`project.md`→project、`phases/<phase>.md`→phase(値はファイル名)。org / team / project はファイル名由来で全ステージに普遍的な既定として適用され、該当 phase ルールはステージ側の `phase: <name>` という**プル型のインポート**で付きます。この「関係をステージ側に持たせる」向きは、`requires_stage` や `consumes` と同じ設計です。

コンパイル出力(`stage-graph.json` の各ステージの `rules_in_context`)が解決済みチェーンを各ノードに焼き込み、**実行時はチェーンを歩きません** — 解決はコンパイルが所有します。衝突はメモリゲート(§13 の admission チェック)で拒否され、これはセクション単位の LLM チェックです。practices-discovery の是認ゲートはもう一つの admission ゲートですが、その昇格は是認によって正当化される決定的なセクション置換(`aidlc-state.ts practices-promote`)で、自動 org 衝突チェックは走りません。この「コンパイルが一度だけ解決し、ランタイムは読むだけ」という設計が、再現可能な実行と、再起動後のクリーンな復旧を生みます — ゲートで承認したときのルール集合が、実行の途中で足元から変わることがないからです。

ルールは `pairing:` フィールドで決定的センサーと対にできます — 値は `feedforward-only`(センサーの相棒を持たない明示)か `<sensor-id>`(既存の `.claude/sensors/aidlc-<id>.md` に一致)です。`/aidlc --doctor` は 2 つの助言行を持ちます — **Rule drift**(org が後から変わり、team/project の見出しが _populated な_ org 見出しと重なる箇所を、ファイル・セクション・引用付きで検出)と **Paired sensor coverage**(`pairing: <sensor-id>` を持つルールの named センサーが少なくとも 1 ステージの解決済みセンサー集合に存在するかを確認する、意味ではなく存在のチェック)。どちらも read-only で常に通り、健全性の終了コードを変えません。ドリフト検出自体はバイト再現可能な見出し・文字列作業で、矛盾の判定は観測時点のオーケストレーター LLM に委ねられます。

フォワード互換の方針も明確です — 新フィールドは加算的に載り、既存のルール行を書き換える必要はなく、消費側は未知の frontmatter キーを許容(無視して通す)します。これにより、レガシーの `enforcement:` / `overrides:` キーがユーザー拡張に残っていてもエラーなく読み込まれます(リゾルバは無視します)。学習ループのメモリゲートは、確認された各学習を `memory/project.md`(既定)や `memory/team.md`(ワンクリック昇格・org スコープ書き込みなし)へ、日記見出しの話題の下に日付入りで直接書きます。リゾルバはそれらをクリーンな整数チェーン `SCOPE_PRIORITY`(org:0・team:1・project:2・phase:3)で並べ、分数の 1.5 / 2.5 段はありません。v0.5.0 より前のルールファイルは、リネーム+フラット化後もそのまま読め、移行はパスのみです。このフォワード互換の姿勢が、チームが自分のルールを書き足しても壊れない基盤になっています。以上をまとめると、v2 のステアリングは「ファイル名でスコープを決める 5 層の strict-additive チェーンを、ワークフロー開始時に一度だけコンパイルして焼き込み、学習ループとセンサーで前送り・後送りの両面から steer する」構造です。次単元では、この単一ソースのルールを各ハーネスがどう読み込むか — ハーネス統合 — へ進みます。

## v1 の2層ルール構造との対比

レガシーの **v1(v1.0.1)** は、まったく異なる構造でした。配布物 `aidlc-rules/` は 2 つのサブディレクトリから成ります — `aws-aidlc-rules/`(コアの AI-DLC ワークフロールール、`core-workflow.md`)と `aws-aidlc-rule-details/`(コアが条件付きで参照する詳細ルール)。これが v1 の**2 層**の骨格です。

v1 のコアワークフローは、フェーズ実行時に詳細ルールファイルを**オンデマンドで**読み込む前提でした。`common/process-overview.md`・`common/session-continuity.md`・`common/question-format-guide.md` などの共通ルールをワークフロー開始時に必ずロードし、`inception/workspace-detection.md` のようなステージ別詳細はそのステージで読みます。ルール詳細ディレクトリの場所は、セットアップ手段(Kiro なら `.kiro/aws-aidlc-rule-details/`、Cursor/Cline/Claude Code/Codex なら `.aidlc-rule-details/`、Amazon Q なら `.amazonq/aws-aidlc-rule-details/`)で変わり、コアが「最初に存在するものを使う」方式で解決していました。プラットフォーム別のセットアップも、Kiro は Steering Files、Amazon Q は Q Rules、Cursor は Cursor Rules、というように各ツールのルール機構へ手動でコピーする形でした。

v1 はまた、アダプティブな性格を「ワークフローが作業に適応する」という原則に置き、コアワークフローが**Inception / Construction / Operations の 3 フェーズ**を統べていました(v2 の 5 フェーズ・32 ステージへの拡張はここが起点です)。コンテンツ検証(Mermaid 図・ASCII 図・特殊文字エスケープ)や質問形式(A〜E 選択肢+`[Answer]:` タグ)、ウェルカムメッセージの表示なども、すべてコアワークフローが `common/` の詳細ルールを参照して強制する MANDATORY 事項として記述されていました。つまり v1 は「1 つの巨大なコアワークフロー+条件付き詳細ルール+opt-in 拡張」という散文駆動の世界だったのです。

v1 にはもう一つ、**extensions(拡張)の opt-in** 機構がありました。ワークフロー開始時に `extensions/` を再帰スキャンして軽量な `*.opt-in.md` だけを先読みし、要件分析でユーザーに opt-in を提示、ユーザーが選んだときに初めて対応する完全なルールファイルを読み込む(選ばなければ読まずコンテキストを節約する)— 有効化された拡張ルールは「ハード制約」であり、適用可能なのに非準拠なら**ブロッキング所見**として扱われ、`aidlc-state.md` の `## Extension Configuration` で有効/無効を切り替えられました。`*.opt-in.md` を持たない拡張は常に強制でした。

対比を整理すると、v1 は「コアワークフロー + 詳細ルール + opt-in 拡張」という 2 層+オンデマンド読み込み+`enforcement`/opt-in の世界でした。v2 はそれを、`memory/` 配下の **5 層 strict-additive チェーン**(org → team → project → phase → stage)へ再設計し、`overrides:`/`enforcement:` を廃し、衝突を書き込み時に捕まえ、**学習ループ**でチームの訂正を永続ルールへ育て、**Sensors** で決定的に裏取りする、という control loop の世界へ進化させました。言い換えると、v1 は「どう振る舞うべきか」を大きな散文とオンデマンドの詳細で指示していたのに対し、v2 は「振る舞いの標準を層ごとに宣言し、実行前にコンパイルして衝突を潰し、実行後にセンサーで裏取りし、人間の訂正を次回への学習として畳み込む」という、より制御工学的な形に整理されています。v1 は正式リリース済みで「v1 を知りたい層」には有用ですが、現行の学習体験の主軸は v2 のこの構造です。移行そのものに専用ガイド doc は無く、各版の CHANGELOG の「Upgrade: re-copy your `dist/<harness>/`」注記で表現されます。
