AI-DLC はチームのニーズに合わせて調整できるよう設計されています。この単元では、settings のオーバーライド・ツール権限・5 層ルールチェーンでのチーム適応・二層の Team Knowledge・学習ループによる自己修正という 5 つのカスタマイズ面を、黒帯として一段深く押さえます。基本の線は一貫して「データ対コード」— 振る舞いの調整はほとんどが Markdown か JSON 設定の編集で済み、TypeScript には触れません。5 節と多めなので各節は要点を絞って進めます。

## settings のオーバーライドとスコープ・深度・テスト戦略

共有の `.claude/settings.json` はフレームワークに同梱されバージョン管理されます。チームに影響を与えず自分の環境だけを変えたいときは、`.claude/settings.local.json`(`.gitignore` 済み)を作ります。用途はモデル選択の上書き(別の Opus / Sonnet モデル ID への切替)・ローカル環境変数・自分のセキュリティ要件に合わせたツール権限調整など。**機微な値(秘密情報・認証情報・個人的な上書き)は、コミットされる `settings.json` には決して置かず、gitignore された `settings.local.json` に置く** のが鉄則です。

プロジェクトの全ワークフローを同じスコープで始めたいとき(例: 参加者全員が `workshop` を走らせるワークショップ)は、`settings.json` の `env` ブロックで `AWS_AIDLC_DEFAULT_SCOPE` を設定します。これで素の `/aidlc` 起動が既定スコープを使い、参加者は毎回 `/aidlc workshop` と打たずに済みます。この環境変数はワークフロー初期化時にだけ読まれ、いったん intent の `aidlc-state.md` ができれば state ファイルが正本になり、走行中のワークフローに env 変更は影響しません。スコープ決定の優先順位は高い順に、(1) 明示 CLI フラグ(`/aidlc feature` や `--scope bugfix`)、(2) フリーテキストのキーワード検出(`/aidlc fix the login bug` → `bugfix`)、(3) `AWS_AIDLC_DEFAULT_SCOPE` 環境変数、(4) ハードコードのフォールバック(intent 誕生時 `poc`・未マッチのフリーテキストは `feature`)です。有効値は enterprise・feature・mvp・poc・bugfix・refactor・infra・security-patch・workshop の 9 つで、無効値は起動時に明確なメッセージでエラーになります。

なぜスコープだけを環境変数化し、深度やテスト戦略はしないのか — 各スコープが自分の深度とテスト戦略の既定を宣言しており(workshop なら Standard 深度・Minimal テスト戦略)、スコープを決めればそれらが連鎖して決まるからです。上書きが要るときだけ `--depth` / `--test-strategy` を CLI で渡します。スコープや深度は承認ゲートでいつでも変更でき、`/aidlc --scope enterprise` のようなユーティリティコマンドでも切り替えられます。設定の確認は `/aidlc --doctor` で行え、`AWS_AIDLC_DEFAULT_SCOPE` が設定され有効かを表示します。env 既定が適用されると、オーケストレーターはワークフロー開始時に一行の通知(`Using scope=<value> from AWS_AIDLC_DEFAULT_SCOPE`)を出すので、スコープの出所が効いた瞬間に見えます。

なお、この章のメカニズム寄りの設定 — `settings.json` / `settings.local.json`・statusline コマンド・`$CLAUDE_PROJECT_DIR`・ツール権限ブロック — は **Claude Code 固有** です。スコープ設定・深度・知識・ルールといったハーネス中立なカスタマイズはどのハーネスでも効きますが、Kiro は `.kiro/settings/cli.json` とエージェント設定で、Codex は `.codex/config.toml` と Starlark ルールで同等物を設定します。statusline も Claude Code だけの機能で、Kiro は `/aidlc --status`、Codex は `update_plan` の task-progress 項目と `$aidlc --status` でワークフロー位置を表面化します。ハーネスをまたぐときは「中立な調整はそのまま効くが、メカニズム設定は各ハーネスの置き場所を見る」と切り分けると迷いません。

## ツール権限の拡張と縮小・Extending AI-DLC

`.claude/settings.json` の `permissions.allow` リストは Claude Code のツールを事前承認し、ワークフローが呼び出しのたびに権限プロンプトを出さずに走るようにします。スコープ付きの `Bash(bun $CLAUDE_PROJECT_DIR/.claude/tools/*)` が素の `Bash` より前に並び、フレームワーク自身のツール呼び出しが常に狭いルールに先にマッチするようになっています。

権限の効き方には 3 つの原則があります。第一に allow リストがプロジェクト全体の **上限(ceiling)** であること。第二にエージェントは既定でセッションのツールセット全部を継承し、唯一の出荷時制限が `disallowedTools: Task`(ネストしたサブエージェント生成の禁止)であること。第三にエージェントの frontmatter に `tools:` 許可リストを足せば **個別に絞れる** こと(ただし `tools:` を列挙すると、完全修飾 `mcp__<server>__<tool>` を併記しない限り MCP ツールは落ちます)。権限を **広げる** のはカスタムステージが追加の能力を要するときだけ。**狭める** と手動承認が必要になります — たとえば `Task` を外すとサブエージェントステージ(2.1 Reverse Engineering・3.5 Code Generation)が委譲のたびにプロンプトを出します(なお 0.2 Workspace Detection は `aidlc-utility init` 内で決定的に走り Task を使いません)。

日々のチューニングを超えて、ステージ追加・エージェント追加・スコープ定義・ルール教育・センサー結線・ドメイン知識追加といった「フレームワーク自体の再形成」をしたくなったら、それは別の仕事で、**Harness Engineer Guide** が入口です。境界線はやはりデータ対コードで、そのガイドの対象はすべて YAML frontmatter 付き Markdown か JSON 設定 — TypeScript 編集はありません。フレームワークの _コード_(オーケストレーター・フック・CLI ツール・コンパイルパイプライン)を変えるなら、それは Developer Reference の領域です。この線引きを意識すると、「今やろうとしている変更はデータか、コードか」で入口を即断できます。

## 5 層ルールチェーンでのチーム/プロジェクト適応

ルールはワークスペースルートの `aidlc/spaces/<space>/memory/` に Markdown で置かれ、`org.md`・`team.md`・`project.md`・`phases/<phase>.md` という **ファイル名でスコープが決まります**(`scope:` フィールドは存在しません)。org はフレームワーク既定、team はチームが affirm した実践、project はこのプロジェクト固有、phases/<phase>.md はそのフェーズの全ステージに効くルールです。これらはワークフロー開始時に `org → team → project → phase → stage` の 5 層チェーンで解決されます。

このモデルは **strict-additive(厳密加算)** です。適用されるルールはすべてエージェントの文脈に現れ、実行時に黙って落ちたり上書きされたりしません — org・team・project は連結され、何も drop しません。旧版にあった `overrides:` ブロックや `enforcement:` キーワードはもう無く、スキーマから削除されています。衝突は実行時ではなく **ルールが書かれる時** に、memory ゲートの admission チェックで弾かれます。チェーンは一度だけ、ワークフロー開始時に解決され、ステージ定義・ルール・センサーが 1 つのグラフ(`stage-graph.json` の各ノードの `rules_in_context`)にコンパイルされます — 実行時にチェーンを歩き直すことはなく、コンパイルが解決を所有します。

phase 層は直交していて、ステージが frontmatter で `phase: <name>` を宣言しているために、対応する `phases/<name>.md` が **pull import** として引き込まれます(glob フィルタも具体パス合成もありません)。stage 層(`aidlc-stage-<slug>.md`)は将来用に予約されています。だからチーム標準は `team.md` に、このプロジェクト固有の逸脱は `project.md` に、フェーズ横断の決まり(例: inception では全アーキ判断に代替案を 2 つ以上記録する)は `phases/inception.md` に書けば、以降の該当ステージの文脈に確実に載る、というのが適応の基本動線です。org 変更が team/project より後に起きて矛盾が生じた場合は、`/aidlc --doctor` の rule-drift 行が、ファイル・セクション・衝突する org の一文を挙げてオンデマンドで知らせます(advisory でブロックはしません)。

ルールはさらに、決定的なセンサーと **ペアリング** できます。frontmatter の `pairing:` に `<sensor-id>`(`.claude/sensors/aidlc-<id>.md` の `id:` と一致)を書けばそのセンサーと組み、`feedforward-only` と書けば「このルールにはセンサーの相棒がない(フレームワークが決定的に検証できない)」と明示します。`/aidlc --doctor` の paired-coverage 行は、`pairing:` を持つルールのうちセンサーが解決できたものを数え(`P/(M-X) guardrails paired`)、未ペアのルールを列挙します。ただしこれは **存在チェックであって意味チェックではありません** — 結線が解決することは確かめても、そのセンサーがそのルールに合っているかは見ません。ルールが「前向きの指示」、センサーが「後ろ向きの検証」で、両者を意識的に対にしておくと、チームの決まりが prose の中だけで空回りせず、実際の出力に対して byte 単位で効いているかを観測できます。旧来の `enforcement: enforced` や `overrides:` は完全に廃され、未知キーは寛容に無視される前方互換方針の下で、ルール frontmatter は加算的にのみ拡張されます。

## Team Knowledge(二層)の追加

Knowledge は「エージェントが働く前に読む」ドメイン文脈で、フレームワークが強制する制約であるルールとは別物です。Knowledge は二層に分かれ、あなたが編集するのは片方だけです。**Tier 1(方法論知識)** は `.claude/knowledge/` に同梱され、各エージェントがステージを走らせるのに使う方法論リファレンス(`aidlc-architect-agent/architecture-guide.md` など)を保持します。アップグレードのたびに上書きされるので **触ってはいけません** — 足したものは次のバージョンで消えます。**Tier 2(チーム知識)** はあなたのもので、Space レベルの `aidlc/knowledge/`(= `aidlc/spaces/<space>/knowledge/`)にあり、`memory/`・`codekb/`・`intents/` の兄弟として space 内の全 intent にまたがって蓄積されます。フレームワークは決して上書きせず、初回 `/aidlc` で空の `aidlc/knowledge/` を作るだけで、中身はあなたが埋めます。

置き場所が読み手を決めます。`aidlc/knowledge/aidlc-shared/` は **全 11 エージェントが毎ステージ** 読み(命名規約・コミット形式・ドメイン用語集など横断的な標準向け)、`aidlc/knowledge/<agent>-agent/` は **そのエージェントがリードのときだけ** 読みます(1 つの役割向けドメイン文脈)。ディレクトリ名はエージェントスラグと完全一致が必須で(`aidlc-architect-agent/` であって `architect/` ではない)、タイプミスはエラーなく黙って無視される最頻の落とし穴です。ステージ開始時の 6 ステップ読み込み順で Tier 2 はステップ 4・5 に入り、ディレクトリが存在しファイルがあるときだけ発火します。読み込みは毎ステージ開始時なので、ファイル編集は次の `/aidlc` で即反映され、キャッシュも再起動もレジストリも要りません。

知識かルールかの判断は「その指示に違反したら人間レビュアーがステージ出力を **却下** するか?」で、却下するならルール、レビュー時の背景として使うだけなら知識です。API 設計標準やドメイン用語集は知識(`aidlc-shared/` や該当エージェントの Tier 2 へ)、「PII は決してログに出さない」「全データアクセスはリポジトリ層を通す」のような非交渉の決まりはルール(`memory/` へ)です。エージェントは知識ファイルを **文字どおり・等しい重み** で読むため、古い・矛盾したファイルは現行と同じ権威で誤誘導します — Tier 2 ツリーはコードのように剪定が要る、と心得ておきます。

チーム知識とルール(方法層)はどちらも **per-space** です。複数チームが 1 プロジェクトを共有するとき、各チームは自分の `aidlc/spaces/<name>/` に `memory/`・`knowledge/`・`codekb/`・`intents/` を同じ形で持ちます。あなたが埋めた `aidlc/knowledge/<agent>-agent/` は 1 つの space の中に住み、別チームは自分の空の `knowledge/` ツリーを埋めるので、ファイルは境界を越えて漏れず、相手のものがあなたのエージェントの文脈を薄めることもありません。方法層(`memory/` の `org.md → team.md → project.md`)もアクティブ space 内で解決され、新しい space はフレームワークのベースライン(`org.md` コピー・空の `team.md`/`project.md`)から seed されるので、新チームは他チームの実践を継承せず自分のものを獲得します。space は `core/` で書くものではなく、インストール済みプロジェクトで `/aidlc space-create <name>` で作るランタイムのチームデータ — ここでもデータ対コードの線がそのまま通っています。

## 学習ループによる自己修正

最後の面は、一度きりの訂正を恒久的なルールに変える **学習ループ** です。ステージ実行中、フレームワークは `<record>/<phase>/<stage>/memory.md` に観測を書き続けます。エントリは 4 つの見出し — Interpretations(曖昧な所での解釈)・Deviations(意図的な逸脱)・Tradeoffs(検討した代替と選択理由)・Open questions(次回までに確認したいこと)— の下に、タイムスタンプ付きで並びます。言語モデルの仕事はこの日記書きだけで、それ以降(数える・提示する・ルーティングする・書く)は決定的ツールとユーザーの選択が担います。

各承認ゲートの前に **学習ゲート(§13 儀式)** が走り、日記から拾った候補(逐語提示。要約も「面白いもの」フィルタもなし)と「次回のために何か追加は?」の自由記述を 1 つの確認リストにして提示します。あなたがチェックしたものだけが残ります。行き先は見出しが決めます — Interpretations・Deviations・Tradeoffs は `memory/project.md` にプラクティスとして着地し(確定した学び **=** プラクティス)、Open questions は研究項目なので昇格しません。既定スコープは project で、ワンクリックの「team に昇格」で `team.md` へ広げられますが、org へ書くパスはありません(最も狭いスコープを既定にして、1 つのプロジェクトの驚きが組織全体のルールに化けるのを防ぎます)。

書き込み前に `org.md` との **セクション単位の衝突チェック** が走り、提案が同じ見出しの org ルールと矛盾すれば、ゲートが止まって衝突する一文をその場で引用し、修正・スキップ・エスカレーションを促します(override パスはありません)。日記・ゲート確認・ファイル書き込みはそれぞれ監査行(`RULE_LEARNED` / `SENSOR_PROPOSED`)を残すので、ルールが黙って入ることはありません。そして重要なのは、捉えた学びが効くのは **今のワークフローではなく次から** だという点 — 走行中は開始時のコンパイル済みビューを保ち、地面を途中で変えない予測可能性を守ります(BGP がパケット飛行中に経路を再計算しないのと同じ)。学びがルールではなく **センサー結線** の場合は、マニフェストの scaffold とステージの `sensors:` への id 追記を原子的に 2 箇所書き込みます。こうしてチーム適応は、設定・権限・ルール層・知識・学習ループという 5 つの面から、コードに触れずデータで形づくられます。具体例で締めましょう — ある銀行プロジェクトで、ステークホルダーの「retry で取引が重複しないように」というメモを、エージェントが DB トランザクションの ACID 要件と解釈してしまう。ユーザーが「これは決済(銀行取引)のこと」と訂正すると、日記が Interpretations にそれを記録し、ゲートで候補として提示され、ユーザーがチェックすれば `memory/project.md` にプラクティスとして着地します。その学びは今の実行ではなく次のワークフローから効き、以降エージェントは「取引 = 決済」と理解して走る — 一度払ったコストが、次からの全ワークフローで回収される、というのが学習ループの実利です。
