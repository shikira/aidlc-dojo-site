Construction でビルドしテストしたソフトウェアは、そのままでは価値を生みません。動かし、監視し、壊れたときに直し、負荷に耐えることを確かめ、そこから学んで次に活かす — この一連が Operation フェーズです。v1 では Operations は実体のない PLACEHOLDER で「すべてのビルドとテストは Construction で扱う」とだけ書かれていましたが、v2 はここを実体のある 7 ステージ(4.1〜4.7)として持ち、最後の 4.7 でループが閉じてまた Ideation へ還ります。デプロイして終わりではなく、運用しながら学び、その学びを次の企画に流し込む — v2 の Operation はこの「回り続けるライフサイクル」を初めて具体化した段です。この単元では、その 7 ステージがどう並び、誰がリードし、何を成果物として出し、どこで継続改善につながるかを、黒帯として一段深く押さえます。Operation は前半のデプロイ経路(4.1〜4.3)、中盤の運用準備(4.4〜4.6)、そして継続改善(4.7)という三つの塊に分かれ、最後だけが 3 択ゲートを持つ、という構造をまず頭に入れておくと全体が見通せます。

## Operation フェーズ概観とデプロイ経路(4.1〜4.3)

Operation は AI-DLC ライフサイクル 5 フェーズの最後です。Construction が生んだ動くコードを、デプロイ・監視・インシデント対応・性能検証・継続最適化へと運びます。7 ステージ(4.1〜4.7)は pipeline 設定・環境プロビジョニング・デプロイ実行・観測性・インシデント対応・性能検証・フィードバック収集をカバーします。

7 ステージはすべて **CONDITIONAL(条件付き)** で、スコープと実行計画によって実行有無が決まります。たとえば mvp・poc・bugfix・refactor スコープは Operation を丸ごとスキップし、infra は 4.1〜4.4(deployment-pipeline・environment-provisioning・deployment-execution・observability-setup)、security-patch は 4.1 と 4.3 のみを走らせます。enterprise・feature・workshop は 7 ステージ全部です。また Operation の全ステージは **inline** で走ります — このフェーズにサブエージェントはいません。承認ゲート・質問形式・完了メッセージ・状態追跡はすべて `stage-protocol.md` の共有パターンに従うため、どのステージも一貫した進め方になります。

前半のデプロイ経路は 3 ステージです。**4.1 Deployment Pipeline** は CD パイプライン設定・デプロイ戦略(blue/green・canary・rolling)・ロールバック手順・環境昇格ゲートを設計し、`cd-config.md` / `deployment-strategy.md` / `rollback-runbook.md` を出します。リード役は aidlc-pipeline-deploy-agent。入力は 3.7 の CI パイプライン設定と 3.4 のインフラ設計で、既存のパイプラインが十分にあればスキップされます。**4.2 Environment Provisioning** は Construction の IaC を使って対象 AWS 環境をプロビジョニングし検証します。ここは複数エージェントで、aidlc-aws-platform-agent がリード、aidlc-devsecops-agent がセキュリティ姿勢を、aidlc-compliance-agent がデータ所在地と規制コントロールを検証し、`environment-inventory.md` と `validation-report.md`(ヘルスチェック含む)を出します。**4.3 Deployment Execution** は実際のデプロイ — アーティファクトをパイプラインに流し、スモークテストとヘルスチェックを走らせ、DB マイグレーションを実行します(aidlc-developer-agent が支援)。成果物は `deployment-log.md`・`smoke-test-results.md`・`health-check-report.md`。この 3 ステージはいずれも承認ゲートが厳密に 2 択(Approve / Request Changes)である点が共通します。複数エージェントのステージでは、コンダクターがまずリードを呼び、その出力を文脈として支援エージェントを呼びます — すべての委譲はコンダクターが行い、エージェント同士は決して呼び合いません。この「リード先行・支援後続・コンダクター一元」の型が、4.2 と 4.3、そして最終の 4.7 に共通する多エージェント運用の基本です。

デプロイ経路の 3 ステージには明確な依存の順があります。4.1 がパイプラインとデプロイ戦略・ロールバック手順を定め、4.2 がそのパイプライン設定とインフラ設計を入力に環境を用意し、4.3 が「パイプラインと環境が整った後」に実デプロイを行う — 前段の成果物が次段の入力になる直列の鎖です。だから 4.1 のロールバックランブックが具体的でなければ 4.3 のデプロイは安全網を欠き、4.2 のセキュリティ・コンプライアンス検証が甘ければ本番環境にリスクを残したままデプロイに進むことになります。Operation の各ステージが CONDITIONAL なのは、既に十分なパイプラインや環境がある場合に重複作業を避けるためで、「もう在るものは作り直さない」という判断がスコープと実行計画に組み込まれています。

## Observability(4.4)と Incident Response(4.5)

デプロイしたら、次は「動いているか」を見えるようにし、「壊れたとき」に備えます。ここが運用準備の中核です。

**4.4 Observability Setup** は Operation で **最も多くの成果物を出す** ステージです。監視・ダッシュボード・アラーム・SLO/SLI 追跡・ログクエリ・分散トレーシング・異常検知を設定し、`dashboards.md`(CloudWatch ダッシュボード)・`alarms.md`(重大度・SNS ルーティング・エスカレーション付きアラーム定義)・`slo-config.md`(SLO/SLI 追跡)・`log-queries.md`(CloudWatch Logs Insights の保存クエリ)・`tracing-config.md`(X-Ray トレーシング)・`anomaly-config.md`(異常検知)という 6 つの内容ファイル(+質問ファイル)を生みます。リードは aidlc-operations-agent。CloudWatch・X-Ray・SNS といった AWS 固有の設計ですが、ダッシュボード・アラーム・SLO・トレーシングというパターン自体は他の基盤にも移せます。入力は 3.3 の NFR 設計・3.4 のインフラ設計・そしてデプロイ済みアプリケーションです。

**4.5 Incident Response & Runbook Generation** は運用ランブック・インシデント対応計画・エスカレーション手順を生成します。成果物は `runbooks.md`(SSM Automation ランブック群)・`incident-plan.md`(AWS Incident Manager 連携のインシデント対応計画)・`escalation-matrix.md`(エスカレーション経路・オンコールローテーション・連絡手順)。入力は 4.4 の観測性設定・3.3 の NFR 設計・3.4 のインフラ設計です。このステージは POC や非本番デプロイでは条件付きでスキップされます。観測性(4.4)を入力に取るので、「見える化」の後に「壊れたときの動き方」を固める、という順序になっています。

観測性とインシデント対応は表裏の関係です。アラームや SLO で異常を検知できて初めて、ランブックとエスカレーション表が意味を持ちます。逆に、検知だけあって対応手順が無ければ、深夜の呼び出しが即興対応になってしまう — だから v2 はこの 2 つを隣り合うステージとして並べています。SLO/SLI という「守るべき水準」を 4.4 で数値として定義し、それを割りそうになったときに 4.5 の手順で動く、という一貫した筋が通っているわけです。運用の質は、この 2 ステージの成果物がどれだけ具体的か — アラーム閾値が SLO 割れの手前に置かれているか、ランブックにエスカレーション経路と連絡先が書かれているか — に大きく依存します。ダッシュボードが飾りに終わらず、アラームが実際にオンコールへ届き、届いた先に手順があって初めて、観測性への投資が回収されます。

観測性の 6 成果物は、それぞれが役割を分担します。ダッシュボードは「今どうなっているか」を人が読める形にし、アラームは「閾値を越えたら誰にどう知らせるか」を定義し、SLO/SLI 設定は「守るべき水準」を数値で固定します。ログクエリ(CloudWatch Logs Insights の保存クエリ)は障害調査の初動を速くし、X-Ray トレーシングは分散したサービス間の遅延やエラーの発生箇所を追え、異常検知は閾値ベースでは捉えにくい変化を拾います。これらが揃うことで、4.5 のインシデント対応が「勘」ではなく「計器」に基づいて動けるようになります。SLO は後段 4.6 の性能検証と 4.7 の SLO 遵守レビュー(エラーバジェット消費率)にもそのまま引き継がれるため、ここで曖昧な水準を置くと下流全体が曖昧になります。観測性が Operation で最多の成果物を出すステージなのは、後続のインシデント対応・性能検証・継続最適化のすべてが、この段で敷いた計器の上に乗るからです。ここで手を抜くと、後段のすべてが土台のない上に積まれることになります。

## Performance Validation(4.6)

**4.6 Performance Validation & Load Testing** は、負荷をかけて NFR の性能目標をデプロイ済みアプリに対して検証するステージです。リードは aidlc-quality-agent。入力は 3.2 の NFR 要件・3.3 の NFR 設計・4.4 の観測データです。POC や性能がクリティカルでないアプリでは条件付きでスキップされます。

成果物は 4 つ。`load-test-plan.md`(シナリオ・ツール・構成を含む負荷テスト計画)、`test-results.md`(レイテンシ・スループット・エラー率などの性能テスト結果)、`nfr-validation-matrix.md`(NFR 目標値 vs 実測値の検証マトリクス)、そして質問ファイルです。

ここでのポイントは「目標 vs 実測」を突き合わせるマトリクスの存在です。Inception/Construction で決めた NFR は仮説にすぎず、実環境で負荷をかけて初めて満たされているかが分かります。目標値を並べただけでは検証になりません — 実測を横に置き、達成/未達を一目で分かる表にすることで、次に何を最適化すべきかが見えます。負荷テスト計画では、想定するトラフィックのシナリオ、使用する負荷ツール、テスト構成を先に固め、そのうえで実行してレイテンシ・スループット・エラー率を測ります。そして 4.4 の観測データを入力に取ることで、性能検証は「テスト専用の計測」ではなく、本番と同じ観測基盤の上で行われます — 計測系が二重にならず、検証で見た数値と日常の監視で見る数値が地続きになる、という設計です。段としては、デプロイ(4.1〜4.3)で「動く」を、観測とインシデント対応(4.4〜4.5)で「見える・直せる」を確立し、この 4.6 で「目標の負荷に耐える」を確かめて、運用準備の三本柱が揃う、という位置づけになります。NFR が実測で裏づけられていない状態で本番トラフィックを迎えるのは、仮説のまま賭けるようなもの — だから性能検証は、観測性が整った後、継続最適化に入る前の関門として置かれています。

負荷テストは容量計画にも直結します。実測したレイテンシ・スループット・エラー率は、想定ピーク時にシステムがどこで頭打ちになるかを示し、スケール方針(水平/垂直)やリソース配分の根拠になります。エラー率が目標を超える負荷点が分かれば、そこが SLO 上のリスク境界であり、4.7 のエラーバジェット管理の出発点にもなります。ここで aidlc-quality-agent がリードするのは偶然ではなく、性能検証は本質的に「品質の検証」だからです — 機能テストが「正しく動くか」を見るのに対し、負荷テストは「目標の条件下でも正しく動き続けるか」を見ます。NFR 検証マトリクスが未達を明示したら、それは失敗ではなく次の最適化対象の特定であり、承認ゲート(2 択)でその扱いを人間が判断します。性能検証を「本番投入前の一度きりのテスト」と捉えず、観測基盤の上で継続的に回せる計測へと育てておくと、負荷特性の変化やリグレッションを早期に捉えられ、4.7 の継続最適化にもそのまま接続できます。

なお、Operation の 7 ステージがどこまで走るかはスコープが決めるという点は、運用計画を立てるうえで実務的に重要です。enterprise・feature・workshop なら 7 ステージすべてを通しますが、infra は 4.1〜4.4 に絞られ、security-patch は 4.1 と 4.3 だけ、mvp・poc・bugfix・refactor に至っては Operation フェーズ自体をスキップします。つまり「本番運用を伴う変更か、それとも試作や局所修正か」という作業の性質が、そのまま運用準備の厚みに反映される設計です。黒帯として、目の前のスコープがどのステージを含むのかを最初に確認しておけば、不要な成果物を作る手戻りも、必要な備えを飛ばす事故も避けられます。

## Continuous Feedback & Optimization(4.7)と継続改善

**4.7 Continuous Feedback & Optimization** は AI-DLC ワークフロー全体の **最終ステージ** です。SLO 遵守レビュー・コスト最適化分析・インフラのドリフト検知・運用インサイト収集を行います。リードは aidlc-operations-agent、コスト最適化とドリフト検知を aidlc-aws-platform-agent が支援します。入力は Operation フェーズの全アーティファクトと本番監視データです。成果物は `slo-report.md`(SLO 遵守・エラーバジェット消費率)・`cost-analysis.md`(Cost Explorer 分析と最適化提案)・`drift-report.md`(AWS Config ドリフト検知・Trusted Advisor レビュー)・`feedback-loop.md`(運用インサイト・改善提案・次の Ideation サイクルへの入力)です。

このステージだけが持つ、**3 択のユニークな承認ゲート** があります。(1) Approve — ワークフロー完了。AI-DLC ライフサイクル全体が終わる。(2) Request Changes — 修正フィードバックを渡す。(3) Start New Ideation Cycle — `feedback-loop.md` のインサイトを新しい Stage 1.1 に還流させる。他の Operation ステージが 2 択なのに対し、4.7 だけが 3 択なのは偶然ではありません。この 3 番目の選択肢こそが AI-DLC の **循環性** の表れで、運用で得た学び(SLO 割れの傾向・コストの無駄・構成ドリフト)が次の企画へと直接つながります。一度きりのデプロイではないシステムは、ここでループを閉じてまた回り始めます。

4.7 の 4 つのレポートは、それぞれ別の観点から「次にどう改善するか」を語ります。`slo-report.md` は 4.4 で定めた SLO の遵守状況とエラーバジェットの消費率を示し、バジェットを使い切りつつあるなら信頼性への投資が必要だと告げます。`cost-analysis.md` は AWS Cost Explorer による分析と最適化提案で、aidlc-aws-platform-agent が支援して無駄なリソースや過剰プロビジョニングを洗い出します。`drift-report.md` は AWS Config によるドリフト検知と Trusted Advisor のレビューで、IaC で定義した構成から実環境が乖離していないかを確かめます。そして `feedback-loop.md` が運用インサイトと改善提案をまとめ、次の Ideation サイクルへの入力になります。SLO・コスト・ドリフトという 3 つの健全性指標を定点観測し、その総和を次の企画に流す — これが「継続最適化」の実体です。

継続改善はステージ 4.7 だけの話ではありません。並行して働くのが **学習ループ** です。ステージ実行中に `<record>/<phase>/<stage>/memory.md` へ観測(Interpretations・Deviations・Tradeoffs・Open questions の 4 見出し)が記録され、承認ゲートの前の儀式でユーザーがどれを残すか確認し、確定した学びは `aidlc/spaces/<space>/memory/project.md`(既定)にプラクティスとして書かれます。ただしその学びが効くのは **次のワークフロー** からで、走行中のワークフローは開始時にコンパイルしたルールビューを保ちます — ルーターが経路をパケットの飛行中に再計算しないのと同じで、AI-DLC もワークフローの途中でルールを再コンパイルしません。捉えた学びはディスク上にあっても、進行中の実行はその地面を変えない、という予測可能性がここでの肝です。センサー(決定的な第二の意見)も、ステージ出力の書き込みごとに走り、必要な見出しの有無や上流アーティファクトへの参照を byte 単位で確認して、ルールという「前向きの指示」に対する「後ろ向きの検証」を返します。運用は「壊れたら直す」に留まらず、SLO レビュー・コスト分析・ドリフト検知・そして学習ループとセンサーを通じて、システムとチームの両方を継続的に鍛える段になっている — v1 で空白だった Operations が、v2 でライフサイクルを閉じる継続改善の場になった、というのがこのフェーズの要点です。

最後に、Operation を「単発の締め作業」と誤解しないことが黒帯としての勘所です。デプロイは終点ではなく、SLO の遵守を測り、コストを見直し、構成のドリフトを直し、負荷への耐性を確かめ続ける「定常状態」の始まりです。4.7 の 3 択ゲートで Start New Ideation Cycle を選べば、`feedback-loop.md` の運用インサイトがそのまま新しい Stage 1.1 の入力になり、企画→要件→設計→実装→運用→企画…という循環が回り続けます。センサーは advisory(助言的)で、失敗しても監査行と詳細ファイルを残すだけでゲートやワークフローを止めませんが、その「決定的な第二の意見」は、ルールという前向きの指示だけでは拾いきれない実際の出力のズレを、後ろ向きに拾い続けます。運用フェーズの成熟度とは、これらの計器とループがどれだけ具体的に回っているか — 数値化された SLO、届くアラーム、手順のあるランブック、実測された性能、そして次サイクルへ流れるフィードバック — の総体だと言えます。
